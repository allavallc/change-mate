## Goal
Add an optional `**Split-from**: HB-XXX` field to the ticket schema. When a bot or PM splits one ticket into multiple, each child carries `Split-from:` referencing the parent. Renderer parses the field and shows it as a small chip near the existing relationship chips.

## Why
A bot mid-claim sometimes realizes a ticket should have been three. Today there's no way to record that lineage in the schema — provenance lives in commit messages or Notes, neither of which is machine-readable. With this field, `git blame` on the new tickets is one click, and the audit trail for decomposition is preserved without conventions in commit messages.

## Done when
- `build_lib.parse_ticket` recognizes `**Split-from**:` and returns `split_from` on the ticket dict (single ID or list, comma-separated)
- Board renders `Split from HB-XXX` as a chip next to existing relationship chips, same styling
- HORDEOFBOTS.md schema reference lists `Split-from:` as an optional field with example
- PM skill prompt mentions `split_from` so it can populate the field when generating decomposed tickets
- `tests/test_parser.py` covers a ticket with `Split-from` set (single + multi)

## Desired output
A user splits HB-005 into HB-080, HB-081, HB-082. Each child shows a `Split from HB-005` chip. The parent's history is one click away.

## Success signals
- When the user later asks "where did HB-080 come from?", the answer is on the card, not buried in `git log`.
- Bots writing decomposition tickets populate the field by default.

## Failure signals
- Field becomes decorative — bots forget to set it, no inverse rendering on the parent, audit value never materializes.
- Mitigation: PM skill defaults to populating it; HB-076 validator does not require it (optional field).

## Tests
- `tests/test_parser.py` adds two cases: ticket with `Split-from: HB-005` and ticket with `Split-from: HB-005, HB-006`
- `tests/test_build.py` adds a case that confirms the chip renders in HTML

## Notes
- Inverse rendering ("Split into HB-080, HB-081" on the parent) is **out of scope** for this ticket. Renderer infers other inverses (Blocks/Blocked-by) at build time; could add for split later as a polish pass if it becomes valuable.
- Field is optional everywhere — legacy tickets without it parse fine.
