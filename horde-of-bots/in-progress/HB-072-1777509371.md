## Goal
Add a `**Verification**:` field that is orthogonal to `Status:`. Allowed values: `bot-claimed | tests-passed | bot-reviewed | human-reviewed`. Status describes workflow stage; Verification describes trust level. Required on done tickets, optional elsewhere.

## Why
Today a bot moves a ticket to `done/` and there's no way to distinguish "the bot says it's done and walked away" from "the bot finished, tests pass, a human eyeballed it and signed off." All done tickets look equally trustworthy. In a world where bots ship the work, this is the bug — humans need to know which "done" they can trust without reading every diff. A separate field gives a glanceable trust signal without changing the workflow folders.

## Done when
- Schema docs in HORDEOFBOTS.md list allowed values: `bot-claimed | tests-passed | bot-reviewed | human-reviewed`
- `build_lib.parse_ticket` recognizes `**Verification**:` and returns `verification` on the ticket dict
- Default behavior: when a bot transitions a ticket to `done/`, it sets `Verification: bot-claimed` (or `tests-passed` if it ran the tests itself)
- Board renders a verification badge on done cards — mono uppercase, short label (e.g., `BOT` / `TESTS` / `BOT-REV` / `HUMAN-REV`), styled distinctly from status. `bot-claimed` is the visually weakest; `human-reviewed` is the strongest.
- HORDEOFBOTS.md guidance section: explains the four states and who sets each
- PM skill prompt updated: when generating done-state tickets, include `Verification: bot-claimed` by default
- HB-076 validator requires `Verification:` field on done/ tickets and rejects invalid values
- `tests/test_parser.py` covers parsing each of the four values + missing field

## Desired output
A user looks at the done column. Tickets with `Verification: human-reviewed` look visibly more verified than `Verification: bot-claimed`. Trust is glanceable. A reviewer working through the pile can prioritize unreviewed tickets at a glance.

## Success signals
- Over time, the proportion of `bot-claimed` → `human-reviewed` transitions reflects actual review cadence.
- Bots downstream of completed work read `Verification:` to decide whether to trust an upstream change.

## Failure signals
- Everything stays at `bot-claimed` forever because no human ever updates the field. The field becomes decorative noise.
- Mitigation: board UI surfaces "unverified" count somewhere visible (e.g., column header subtitle); makes the gap visible.

## Tests
- `tests/test_parser.py` — parse each of the four values + a missing-field case
- Validator test (in HB-076): done ticket missing `Verification:` fails; done ticket with invalid value fails

## Notes
- **Not** a Status replacement. Status = workflow stage (which folder). Verification = trust level on the contents of a done ticket.
- Allowed values are deliberately small (4) — easy to remember, no ambiguity. Resist adding more.
- Out of scope: a "verifying agent" workflow that automates promotion from `tests-passed` to `bot-reviewed`. That's a v3.0 platform concern; here we're just shipping the schema field so it exists when that work happens.
