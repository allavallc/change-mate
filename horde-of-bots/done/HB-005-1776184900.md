## Goal
Create the Supabase Postgres schema that powers the next-generation board: atomic locking, event history, and per-person write keys — all protected by Row Level Security.

## Why
Current state: locking lives in a GitHub Gist (fragile, not atomic), there is zero history, and there is no access control. This schema is the foundation every other piece of the next build sits on — nothing else ships without it.

## Done when
- `locks` table: `ticket_id` (PK), `agent`, `started_at`, atomic insert-or-fail semantics
- `ticket_events` table: `id`, `ticket_id`, `from_status`, `to_status`, `actor`, `created_at`
- `write_keys` table: `key_hash` (PK), `label`, `role` (`human` | `agent`), `created_at`, `revoked_at`
- RLS policies: anon role read-only on `ticket_events`; write-key-gated inserts on `locks` and `ticket_events`; `write_keys` readable only by service role
- SQL migration script committed to repo under `supabase/migrations/`
- Idempotent: re-running the script against an already-migrated database is a no-op

## Desired output
A single SQL file a maintainer can run in the Supabase SQL editor to provision the project, plus a documented "verify install" check.

## Success signals
- Running the SQL twice produces no errors
- `select` on `write_keys` from anon role returns zero rows (RLS works)
- Inserting a duplicate `ticket_id` into `locks` fails atomically
- `ticket_events` rows visible to anon read, not writable without key

## Failure signals
- RLS disabled accidentally → full-table leak of keys
- Missing index on `ticket_events.ticket_id` → slow queries as history grows
- Lock table not using a unique constraint → two agents both "winning" the same ticket

## Tests
- pgTap or SQL assertion file covering each RLS policy
- Integration test: two concurrent inserts into `locks` for same ticket → exactly one succeeds

## Notes
**Sequencing**: first in the Supabase upgrade chain. Nothing else (HB-006 through HB-009) can land until this is live.

**Alternatives considered**:
- Roll our own auth on top of `horde-of-bots-config.json` PATs → rejected; reinvents what RLS already does and keeps PATs in browser
- Use Supabase Auth (full sign-in flow) → rejected for v1; write-key + RLS is the lighter wedge

- `supabase/tests/verify.sql` + `supabase/tests/concurrent_lock_test.sql` — deep schema assertions + atomic-claim proof, paste in Supabase SQL editor
- `scripts/verify_supabase.py` — one-command live anon-key probe with PGRST002 retry + cold-project diagnosis
