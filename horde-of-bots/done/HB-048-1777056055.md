## Goal
Rewrite the top of HORDEOFBOTS.md so any reader (human or LLM) finishes the first screen knowing Horde of Bots is local dev tooling that must never enter a deploy artifact, code must not depend on Horde of Bots paths, git is the intended sync layer, and Horde of Bots files should be excluded from code-quality passes.

## Why
An adopter using Horde of Bots inside hana-core (a production repo) reported that both humans and LLMs in the repo could not tell whether `horde-of-bots-board.html` was a deployable asset or a dev artifact, so Horde of Bots files drifted into WIP branches and caused deploy-pipeline worry. The feedback author's own TL;DR: "one sentence at the top of HORDEOFBOTS.md saying this is local dev tooling, not product, and must never deploy" is the single highest-leverage change available — everything else in this feature set flows from that.

## Done when
- HORDEOFBOTS.md has a new "Scope" section at the very top (before anything else) stating Horde of Bots is dev-only tooling and must never appear in a deploy artifact.
- HORDEOFBOTS.md has a new "One-way dependency" rule stating Horde of Bots knows about code but code must not reference Horde of Bots paths or tickets.
- HORDEOFBOTS.md has a new "Sync modes" note stating git-sync is the default and documenting that adopters who want Horde of Bots private per-machine can add `horde-of-bots/` to `.gitignore` (with an explicit call-out that this disables collaboration).
- HORDEOFBOTS.md has a new "LLM guidance" block: do not lint/typecheck/format Horde of Bots files as part of code-quality passes, exclude Horde of Bots changes from repo-change summaries by default, assume `horde-of-bots/` does not exist at deploy/build time.
- README.md "Files" table updated to tag each Horde of Bots file as "dev-only tooling".
- No code change. Docs only.

## Desired output
A reader opening HORDEOFBOTS.md for the first time finishes the first screen knowing: what Horde of Bots is, where it must not appear, how code and Horde of Bots relate, whether they can run it privately, and how to treat Horde of Bots files during code review. No inference required.

## Success signals
- Future adopter feedback does not repeat the "is this product code?" confusion.
- LLMs (Claude, Codex, other agents) reading the repo do not propose wiring code references to Horde of Bots paths when drafting feature work.
- Solo-only users no longer ask whether they should commit `horde-of-bots/` to a shared remote.

## Failure signals
- Scope statement drifts out of sync with actual behavior (e.g. a future ticket ships something that does deploy Horde of Bots files — then the statement is a lie).
- "Do not lint/typecheck" guidance gets over-interpreted and adopters exclude Horde of Bots from ALL tooling including basic repo hygiene (trailing whitespace, etc.).
- Local-only `.gitignore` guidance causes adopters to skip commits and then discover their tickets vanished on machine switch.

## Tests
- Manual: read the first two screens of the updated HORDEOFBOTS.md from cold and verify scope, dependency rule, sync modes, and LLM guidance are present and unambiguous.
- Manual: verify README.md files table labels Horde of Bots files as dev-only.
- Existing pytest suite passes unchanged.

## Notes
- **Alternative considered**: split into four tiny tickets (one per feedback item). Rejected — all four changes touch the same HORDEOFBOTS.md preamble, splitting just creates edit churn and four board rebuilds.
- **Alternative considered (sync modes)**: build first-class support for local-only mode (no git sync, live-lock and Realtime still work per-user). Rejected as out of scope — the lock/Realtime design assumes shared ticket state. One-paragraph doc note is correct for v1; revisit if there's real demand.
- **Out of scope**: folder layout changes (HB-049), setup.sh deploy-ignore defaults (HB-050), workflow changes (HB-051).
- **Risk**: LLM guidance is advisory only — adopters' own linters will still run on `horde-of-bots/` unless they exclude it themselves. HB-050 handles the mechanical exclusion.
- **Confidence**: 95%. Highest-ROI ticket in this feature set per the feedback author's own TL;DR.

## Resolution
Added three new top-of-file sections to `HORDEOFBOTS.md`: `## Scope` (scope statement as blockquote + one-way dependency rule), `## Sync modes` (git-sync default + local-only caveat), and `## For LLMs reading this file` (four bullet rules on lint/summary/build-time/ticket-ID treatment). Annotated the `README.md` Files table with `*(dev-only tooling)*` on each Horde of Bots row. Full 77-test pytest suite passes unchanged. No code touched — docs only, as scoped.
