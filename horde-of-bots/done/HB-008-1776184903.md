# [HB-008] Migrate agent locking + event logging from Gist to Supabase

- **Status**: done
- **Priority**: High
- **Effort**: M
- **Feature set**: feature-set-001-supabase-upgrade
- **Assigned to**: allavallc
- **Started**: 2026-04-15
- **Completed**: 2026-04-16

## Goal
Rewrite the agent locking and event-broadcast sections of `HORDEOFBOTS.md` and `build.sh` so that agents lock tickets against the Supabase `locks` table and write status changes into `ticket_events` — retiring the GitHub Gist lock registry.

## Why
The Gist-based lock registry is fragile (race conditions on PATCH), requires every agent to carry a GitHub token with Gist scope, and records no history. Supabase gives us atomic locking and a permanent audit trail in one move — and it removes one more token from the agent environment.

## Done when
- `HORDEOFBOTS.md` Live Lock Registry section rewritten to describe Supabase lock flow
- Agents acquire a lock via insert into `locks` table (fails if row exists)
- Agents release the lock via delete on ticket completion
- Every status move (backlog → in-progress, → done, → blocked, → not-doing) inserts a row into `ticket_events`
- `HORDEOFBOTS_GITHUB_TOKEN` references removed from `HORDEOFBOTS.md`
- `horde-of-bots-config.json` `gist_id` field deprecated (leave for backcompat but document as unused)
- `build.sh` continues to function — board rebuild path unchanged

## Desired output
Agents use Supabase for locking and history; the Gist path is gone; no agent needs `HORDEOFBOTS_GITHUB_TOKEN` anymore.

## Success signals
- Two agents racing for the same ticket → exactly one gets it, other sees a clean "already claimed" message
- `ticket_events` accumulates one row per move
- No agent needs `HORDEOFBOTS_GITHUB_TOKEN` anymore

## Failure signals
- Stale locks when an agent crashes mid-work (need a TTL or manual clear path)
- `ticket_events` double-writes from build-time + runtime
- Agents falling back to "no lock" path silently when Supabase is unreachable (should be a visible warning)

## Tests
- Integration test: parallel agents racing same ticket → one wins, one gets clean error
- Unit test: event row shape matches schema for each status transition
- Manual: simulate Supabase outage → agent behavior is clearly communicated, not silent

## Notes
**Sequencing**: depends on HB-005. Can land in parallel with HB-006/HB-007 since it touches a different surface (agent workflow, not board UI).

**Alternatives considered**:
- Keep Gist locking, add Supabase only for events → rejected; doesn't fix the atomicity bug or the token requirement
- File-based lock in the repo itself → rejected; git push race conditions are worse than Gist races

## Resolution (2026-04-16)
Shipped. Gist locking retired, Supabase is now the lock + event source of truth.

- `locks` table (migration `0001_initial.sql`) with atomic insert-or-fail; SELECT policy for anon added in `0003_locks_select_policy.sql`.
- `ticket_events` rows written by `cm-write` Edge Function on every create; agent-driven status moves insert events directly.
- `HORDEOFBOTS_GITHUB_TOKEN` references removed from `HORDEOFBOTS.md`.
- `gist_id` field in `horde-of-bots-config.json` preserved for backcompat but no longer read.
- `build.sh` board rebuild path unchanged.

Auth context: agent locking now happens in the context of GitHub-token auth (per HB-006 resolution), not write-keys.
