## Goal
Make `setup.sh` safe to run via `curl -fsSL ... | bash` and from CI — the legacy-migration prompt currently uses `read -r -p` which hangs forever on a non-TTY stdin.

## Why
The canonical install instruction in the README is `curl … | bash`. Today, if any of the legacy root-level files happen to exist, that exact command silently hangs because `read` has no TTY. CI installers, devcontainer post-create scripts, and any automation hits this. It's a 5-minute fix that removes a real footgun.

## Done when
- `setup.sh` detects non-interactive stdin (`[ ! -t 0 ]`) and skips the migration prompt with a clear log line: `~ legacy layout detected but stdin is not a TTY — re-run interactively or set CHANGEMATE_AUTO_MIGRATE=yes`.
- `CHANGEMATE_AUTO_MIGRATE=yes` env var (or `--yes` flag, pick one — env var is simpler with `curl|bash`) bypasses the prompt and runs the migration directly.
- `CHANGEMATE_AUTO_MIGRATE=no` env var (or default when stdin not a TTY and var not set) skips migration and proceeds with the rest of the install.
- The interactive prompt path stays the default for direct `bash setup.sh` runs.

## Desired output
`curl -fsSL …/setup.sh | CHANGEMATE_AUTO_MIGRATE=yes bash` works cleanly even on a repo with legacy layout. Direct `bash setup.sh` keeps prompting.

## Success signals
- CI installs of change-mate work without modification.
- Adopters using `curl|bash` no longer see the script appear to hang.

## Failure signals
- Adopter sets `CHANGEMATE_AUTO_MIGRATE=yes` accidentally on a repo that shouldn't migrate. Mitigation: log every move and tell the adopter how to revert (`git checkout -- .`).
- Detection logic gets fooled by some shells that always report stdin as a TTY. Edge case; document the env-var override as the universal fix.

## Tests
- Manual: `bash setup.sh < /dev/null` on a repo with legacy files — confirm no hang, clear log line.
- Manual: `CHANGEMATE_AUTO_MIGRATE=yes bash setup.sh < /dev/null` — migration runs.
- Manual: `bash setup.sh` in a terminal — interactive prompt still appears.

## Notes
- **Alternative considered**: drop the migration entirely from setup.sh and ship a separate `change-mate-migrate.sh`. Rejected — splits the install into two commands for a one-time legacy detection.
- **Out of scope**: changing the migration logic itself; only its triggering UX.
- **Confidence**: 95%.
