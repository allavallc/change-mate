## Goal
Let a user narrow or rank tickets on the board by priority, effort, or feature set without scanning every column.

## Why
As the backlog grows, scanning four columns to find "the highest-priority unblocked thing" or "the smallest-effort win" gets tedious. The current 4-column kanban renders every active ticket — there's no way to focus. A user watching the board needs to slice it fast.

## Done when
- A filter/sort bar appears above the kanban (Board view only — Feature Sets view unchanged).
- Three single-select filter dropdowns: **Priority** (All / Critical / High / Medium / Low), **Effort** (All / XS / S / M / L / XL), **Feature set** (All / each existing fs slug). "All" = no filter.
- One sort dropdown: Default / Priority high → low / Priority low → high / Effort small → large / Effort large → small.
- Default state: every filter "All", sort "Default" — board renders identically to today.
- A "Clear" pill appears when any filter or non-default sort is active; clicking it resets all controls.
- Selections persist in `localStorage` under a namespaced key (`cm_board_filters_v1`); reload preserves them.
- Filters apply to all four kanban columns plus the rejected column when shown.
- Empty columns post-filter still render the "No tickets" placeholder (do not collapse).
- Existing CM-relationship chips and the perimeter-walking robot still render on filtered cards.
- Sort applies within each column independently.
- All logic runs client-side at render time — no rebuild needed for filter/sort changes.
- Mobile: filter bar wraps to multiple rows below 560px without overflow.

## Desired output
User picks Sort = "Priority high → low" and Priority filter = "Critical". The board shows only Critical tickets, ranked by priority, in the columns where they exist. Reload preserves the selection. "Clear" returns to default.

## Success signals
- Time to find the highest-priority unblocked ticket drops from "scan all 4 columns" to one glance.
- Users with > 20 tickets actively use the feature-set filter.
- No console errors, no perceivable lag with 100+ tickets.

## Failure signals
- Filtering hides a ticket the user expected to see (logic bug — likely a missing field default).
- localStorage namespace collides with `cm_github_token` or future state → corruption. Mitigation: clearly namespaced key.
- Filter bar layout breaks at narrow widths.
- Sort isn't stable when two tickets share the same priority/effort — they shuffle on every render. Mitigation: tie-break on original index.

## Tests
- Sort orders Critical > High > Medium > Low and reverse.
- Sort orders XS < S < M < L < XL (small→large) and reverse.
- Filter "Priority = Critical" hides every non-Critical card across columns.
- Filter "Feature set = feature-set-006-redesign" shows only that set's tickets.
- Combined filter + sort renders correctly (e.g. Priority=High, sort by Effort small→large).
- localStorage round-trip: set filters, reload, state preserved.
- Clear button resets all controls and storage.
- Manual: viewport ≤ 560px — filter bar wraps cleanly.
- Manual: in-progress tickets keep the perimeter-walking robot when filtered to visible.

## Notes
- **Alternatives considered — per-column filters**: rejected. The user asked for a single bar; per-column UI multiplies surface area for the same task without proportional benefit.
- **Alternatives considered — header-click-to-sort**: rejected. Discoverable only by accident, doesn't extend to filtering.
- **Out of scope**: search-by-text on title/body. Pull a separate ticket if the user wants it.
- **Out of scope**: filter by assignee. Separate ticket if pulled.
- **Out of scope**: saved filter presets ("my favorite filter"). Over-engineering for v1.
- **Risk**: the poll loop re-renders the board on origin SHA change; filter/sort must apply in the same render pass so the user doesn't see a flash of unfiltered cards. Solution: render order = parse → filter → sort → emit HTML, all in one pass.
- **Confidence**: 85%. The logic is straightforward client-side rendering; the only fiddly bit is keeping the sort stable when ties exist (tie-break on the array index).

## Resolution

Shipped in two phases against `change-mate/build.sh`:

- **Phase 1** (commit `c1559ea`): Filter/sort UI shell. Bar above the board with single-select dropdowns for Priority / Effort / Feature set, sort dropdown, Clear button. Brutalist styling matching CM-060. Bundled with a card layout pass: feature-set chip moved to top, CM-ID `white-space: nowrap`, crab worker moved from top-right to a bottom-left `card-footer`, duplicate `.card-assignee` text removed.
- **Phase 2** (this commit): Wired the actual logic. `FILTER_STATE` lives in `localStorage` under `cm_board_filters_v1`; `loadFilterState/saveFilterState` round-trip; `passesFilter` AND-combines the three filters; `sortTickets` applies stable sort with original-index tie-break. `render()` runs `filterAndSort` per column. Clear button visibility tracks `isFilterDefault()`. `Show rejected` toggle visibility uses pre-filter count so users can still reach the rejected column when filters hide its content.

**Gap surfaced**: no automated browser-side tests for the filter/sort logic — pytest covers parser + build but the JS runs in the browser. Adding jsdom or playwright is its own ticket if pulled. Verified manually: each filter alone, AND combinations, all sort orders, localStorage persistence, Clear reset, mobile wrap.
