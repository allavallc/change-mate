## Goal
Write end-to-end setup documentation so a new project can install Horde of Bots v5 (Supabase locks + events + write keys + Edge Function) from scratch in under 15 minutes.

## Why
The new architecture touches four systems (Supabase Postgres, Supabase Edge Functions, GitHub API, GitHub Actions). Without a clean setup guide, every new install becomes a support ticket. Docs are the final mile that makes HB-005 through HB-008 actually usable by anyone other than us.

## Done when
- `docs/SETUP.md` (or README section) covers:
  - Creating a Supabase project
  - Running the schema migration
  - Deploying the `cm-write` Edge Function
  - Creating and distributing write keys
  - Setting `board_mode` and Supabase credentials in `horde-of-bots-config.json`
  - Configuring GitHub Actions secrets
- Troubleshooting section: common errors (invalid key, stale lock, RLS misconfigured)
- A "verify install" checklist the user can walk through after setup
- `setup.sh` updated to scaffold the new config fields

## Desired output
One markdown file that takes a new user from empty repo → working board with locking and access control.

## Success signals
- A person unfamiliar with the project completes setup from the doc alone
- "Verify install" checklist catches misconfigured RLS and missing secrets

## Failure signals
- Docs drifting from the actual code (pin examples to real file names and commit SHAs)
- Setup steps that assume prior Supabase knowledge without linking to their docs
- Missing coverage of the write-key lifecycle (create / distribute / revoke)

## Tests
- Dogfood: run through the setup doc against a fresh Supabase project — time it, note every friction point
- Doc-lint: all referenced file paths exist in the repo

## Notes
**Sequencing**: last in the chain. Depends on HB-005, HB-006, HB-007, HB-008 all being live.

**Alternatives considered**:
- Defer docs and rely on code reading → rejected; new users are exactly the people who can't read four-system code
- Video walkthrough → rejected for v1; videos rot faster than markdown when files change

## Resolution (2026-04-16)
Shipped via `README.md` + `SETUP.md` (commit `754ad4f` "docs: update everything to reflect GitHub token auth" and follow-ups `3f727cc` "docs: reframe for multi-agent workflows").

Docs cover:
- Supabase project provisioning + migration application
- Deploying `cm-write` Edge Function + setting `GITHUB_PAT`/`GITHUB_OWNER`/`GITHUB_REPO` secrets
- GitHub Actions secrets for board rebuild
- Using a fine-grained GitHub PAT (contents:write) as the write credential
- `setup.sh` scaffolds `horde-of-bots-config.json` with Supabase URL + publishable key

Superseded from original spec:
- "Creating and distributing write keys" — N/A under GitHub-token auth
- `board_mode` — field removed, board visibility follows repo visibility

Troubleshooting + verify-install checklist are lighter than originally specced but present. Can extend when real-world friction surfaces.
