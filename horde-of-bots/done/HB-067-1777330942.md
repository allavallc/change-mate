## Goal
Make it explicit that change-mate tickets must not contain secrets. The agent should refuse to put credentials, keys, passwords, tokens, PII, or internal connection strings into a ticket — and adopters should know the same rule applies on their end.

## Why
Tickets are committed to git, world-readable on public repos, and the agent doesn't redact anything before committing. The reviewer's feedback flagged that internal infrastructure detail (env var names, internal URLs, ports) might end up in tickets. A single rule line in CHANGEMATE.md (so the agent reads it every session) plus a reinforcing paragraph in INSTALL-FAQ.md (so adopters see it on first install) closes that gap.

## Done when
- CHANGEMATE.md → Rules section has a `**No secrets in tickets.**` bullet describing what is forbidden, where it applies (body, Notes, Resolution, commit messages), and how to reference secrets abstractly.
- INSTALL-FAQ.md → Public-repo ticket exposure section has a paragraph reinforcing: even on private repos, treat tickets as untrusted storage; git history persists, repos flip public, contributors get added.

## Resolution
Added the bullet to CHANGEMATE.md Rules and a reinforcing paragraph to the Public-repo ticket exposure section of INSTALL-FAQ.md. Both reference the abstract-vault-path pattern so adopters and the agent have the same model.
