## Goal
Make `bash setup.sh` actually deliver a working board on a fresh repo. Today it only downloads `CHANGEMATE.md` and creates folders — `build.sh`, `build_lib.py`, `change-mate/config.json`, and the GitHub Actions workflow are missing. Adopters who follow the README step `bash change-mate/build.sh` get a "no such file" error.

## Why
The first-time adopter cannot generate a board after running setup.sh. They either copy files manually from the upstream repo (undocumented and brittle) or wait for CI — but the workflow itself isn't installed either, so CI never fires. The install is half-complete by design and that's the single biggest blocker to onboarding.

## Done when
- `setup.sh` downloads the following files into the adopter's repo on a fresh install (creating parent dirs as needed):
  - `change-mate/build.sh`
  - `change-mate/build_lib.py`
  - `change-mate/config.json` (only if not present — don't overwrite an existing one)
  - `change-mate/INSTALL-FAQ.md` (HB-066 ships the content)
  - `.github/workflows/change-mate-rebuild-board.yml`
- A starter `change-mate/board.html` is generated by running `bash change-mate/build.sh` once at end of install (so the adopter sees a board immediately, even before their first push).
- All download URLs use `$REPO_URL/...` so the install honors the same upstream as the existing CHANGEMATE.md fetch.
- Existing files are skipped with the same `~ already present` log line setup.sh uses today — never overwrite.
- The legacy-migration branch (root → `change-mate/`) keeps working unchanged.
- `setup.sh` runs cleanly via `curl -fsSL ...|bash` and via direct invocation. CI smoke-test on a clean tmp dir confirms all expected files appear.

## Desired output
A new adopter runs `bash setup.sh`, commits + pushes, and the GitHub Actions workflow rebuilds the board on the first push because the workflow YAML is now in place. Nothing else they have to do.

## Success signals
- Time-to-working-board drops from "manual copy + 5 follow-up questions" to "two commands."
- Zero "I ran setup.sh but build.sh isn't there" issues post-ship.

## Failure signals
- Workflow YAML overwrites an adopter's existing GitHub Actions setup. Mitigation: skip if file exists, log the skip.
- Generated board.html commits with embedded creds. Mitigation: build.sh no longer touches Supabase, so no creds — but verify.
- Adopter on a flaky network sees a partial install. Mitigation: setup.sh exits non-zero on any download failure (`set -e` already does this).

## Tests
- Manual: `mktemp -d`, `git init`, `bash setup.sh`, confirm all expected files exist.
- Manual: re-run setup.sh in the same dir, confirm no overwrites and no duplicates.
- Manual: in a repo with an unrelated `.github/workflows/` file, confirm setup doesn't touch it.

## Notes
- **Alternative considered**: distribute change-mate as a single `setup.sh` that contains the file content inline (here-docs). Rejected — drift between change-mate's source repo and the inlined version is harder to maintain than `curl` from the source repo.
- **Out of scope**: skill versioning (HB-065), non-interactive prompts (HB-062), README copy (HB-064), FAQ content (HB-066).
- **Risk**: silent rate-limiting on raw.githubusercontent.com if many adopters install at once. Acceptable — it's not a credentialed call and unauth'd raw.githubusercontent has high limits.
- **Confidence**: 90%.
