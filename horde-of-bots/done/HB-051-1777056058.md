## Goal
Stop the board-rebuild workflow from silently producing noise commits on `main` for solo adopters while keeping the "humans watch a fresh Pages-hosted board" primitive working for teams. Default is mode-aware: Supabase configured → auto-commit on; no Supabase → off. Config flag overrides. Rename the workflow so its purpose is self-evident in the GitHub Actions UI.

## Why
Adopter feedback from hana-core: "the auto-rebuild created noise commits in git log that made it hard to see real product changes." A flat default of `false` would kill the noise but break the Pages-fresh-board experience for teams. A flat default of `true` keeps the status quo noise. Mode-aware splits correctly: solo users (no Supabase, usually open the board locally) get a quiet log; teams (Supabase-configured, usually Pages-hosted) stay fresh automatically. Explicit config always wins, so anyone in the edge case (solo on Pages, team viewing locally) has a one-line escape.

## Done when
- `horde-of-bots/config.json` accepts a new optional field `auto_commit_board`: boolean.
- Default is **mode-aware**: when `supabase_url` is present and non-empty in the config, the workflow treats the flag as `true`; when absent or empty string, treats as `false`. An explicit value always wins over the default.
- Workflow reads the flag using `jq` against the config file and only runs `git add / commit / push` when the resolved value is `true`.
- When resolved to `false`, the workflow still **builds** the board (for CI signal and to surface build failures) — it just does not commit it.
- Workflow file renamed: `.github/workflows/build-board.yml` → `.github/workflows/Horde of Bots-rebuild-board.yml`.
- Workflow `name:` field updated to `Horde of Bots: rebuild board` so it's clearly namespaced in the Actions UI.
- HORDEOFBOTS.md "Board rules" section documents the flag, mode-aware default, and how to override.
- README.md / SETUP.md updated with the same.
- Release note explicitly calls out the behavior change for anyone using the workflow today.

## Desired output
Team adopters on GitHub Pages see their board stay fresh automatically (status quo preserved). Solo adopters stop seeing `chore: rebuild board [skip ci]` noise in `git log`. Either group can override with `"auto_commit_board": true/false` if their setup doesn't match the default.

## Success signals
- Noise commits from Horde of Bots on `main` drop to near-zero for solo adopters using default config.
- Team adopters report no change in board freshness.
- No adopter needs to edit the workflow YAML to change behavior — config flag is the sole surface.

## Failure signals
- Solo adopter hosting on Pages sees their board go stale without understanding why. Mitigation: SETUP.md solo-mode section mentions the override.
- Mode detection misfires — e.g. `supabase_url: ""` reads as present and triggers auto-commit. Treat empty string as absent.
- Config key typo (`auto-commit-board` vs `auto_commit_board`) silently takes the default and looks like the flag doesn't work.
- `jq` missing on a minimal runner. GitHub-hosted runners have it — verify explicitly.

- Workflow test: explicit `auto_commit_board: true` + no Supabase → commit.

## Notes
- **Alternative considered**: flat default `false` for everyone. Rejected — breaks the "humans watch" primitive for Pages-hosted team boards until they find the flag.
- **Alternative considered**: flat default `true` for everyone. Rejected — status quo; feedback is that the noise is a real problem for solo users.
- **Alternative considered**: emit the board via Actions deploy to Pages instead of committing to main. Cleaner for Pages adopters but bigger scope — own ticket if pulled for.
- **Out of scope**: rewriting build.sh (HB-049), deploy-ignore defaults (HB-050).
- **Risk**: mode detection uses `supabase_url` as a proxy for "team mode." Adopters with Supabase configured but no Pages get auto-commit they don't need — one-line config fix.
- **Confidence**: 85% (up from 70% with the mode-aware default — solves the Pages-freshness concern surfaced during drafting).

## Resolution
Renamed the workflow file `.github/workflows/build-board.yml` → `.github/workflows/Horde of Bots-rebuild-board.yml` and updated the `name:` field to `Horde of Bots rebuild board`. Split the single commit step into two jobs: a **Resolve auto_commit_board** step that uses `jq` against `horde-of-bots/config.json` to resolve the flag (explicit wins; mode-aware default otherwise — `supabase_url` present and non-empty → true, absent/empty → false) and exports `should_commit` as a step output; and a **Commit and push updated board** step guarded by `if: steps.resolve.outputs.should_commit == 'true'` that runs the original git commands. When resolved false the build step still runs (preserving CI failure signal) but no commit lands. Documented the behavior and the override in `horde-of-bots/HORDEOFBOTS.md` (new "Auto-commit of the generated board" subsection under Board rules), `SETUP.md` (solo-on-Pages note in Step 2), and the project `CLAUDE.md`. Workflow YAML validates cleanly; 77-test pytest suite unchanged and passing. This repo (Horde of Bots itself) has `supabase_url` configured → mode-aware default resolves to `true` → auto-commit continues as before. No behavior change for this repo; solo adopters without Supabase will see the quiet default.
