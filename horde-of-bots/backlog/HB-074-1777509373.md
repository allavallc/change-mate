## Goal
Document and adopt a commit-message convention for bot commits that captures model + trigger + ticket. Pure docs + PM skill update — no schema change, no validator, no enforcement. Audit comes from `git log` once the convention is in place.

## Why
Git already records author, timestamp, and diff. What it doesn't capture: which model/agent did the work, what triggered the commit (claim, heartbeat, done, edit), and which ticket it serves. A simple trailer convention turns `git log --grep "HB-072"` into a complete audit trail with zero new infrastructure. Bots can read each other's history. Humans can answer "what did the bots actually do this week?" without reading source.

The trailer format:

```
HB-072: tests passing on verification field

<commit body — short description of what changed>

Model: claude-opus-4-7
Trigger: HB-072 done
Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>
```

`Trigger:` format: `HB-XXX <action>` where action is one of: `claim`, `heartbeat`, `done`, `edit`, `blocked`, `reclaim`.

## Done when
- HORDEOFBOTS.md gains a "Provenance" section that documents the convention with two or three example commits
- README mentions the convention briefly under an "Auditability" or "Workflow" subsection
- PM skill prompt is updated: when generating commit messages for ticket-related actions, include `Model:` and `Trigger:` trailers in addition to the existing `Co-Authored-By:` trailer
- One pytest assertion in `tests/test_parser.py` (or new `test_pm_skill_format.py`) confirms the PM skill's documented commit format example contains both trailers
- No code changes to build, no schema, no enforcement

## Desired output
Six months from now, `git log` is a readable, machine-greppable record of every bot decision. `git log --grep "Trigger: HB-072"` returns the full lifecycle of HB-072: claim, heartbeats, done, follow-up edits.

## Success signals
- Bots adopt the convention without prompting because the PM skill writes it.
- Humans can audit a week's worth of bot work with two or three `git log` queries.

## Failure signals
- Bots forget the trailers. Audit gracefully degrades — the commit still has author + timestamp + ticket reference in the subject line, just no model/trigger detail.
- Mitigation: convention is documented in PM skill prompts so bots populate by default. If we later see widespread non-compliance, escalate to a commit-msg hook (out of scope for this ticket).

## Tests
- Pytest assertion that the PM skill commit-format example in `skills/product-manager/SKILL.md` contains `Model:` and `Trigger:` lines
- (Optional) doc-lint test that HORDEOFBOTS.md's Provenance section examples include both trailers

## Notes
- **Convention, not enforcement.** Adding a hook to enforce trailer presence would force every contributor to install the hook — adding setup burden the project has explicitly resisted. If bots forget, audit trail thins out but nothing breaks.
- **Why not a `Provenance:` field on the ticket itself?** Provenance is per-commit, not per-ticket; tickets accumulate many commits. Putting it on the ticket would either store stale information (last commit only) or duplicate `git log`. The git layer is the right home.
