## Goal
Make the board update from GitHub directly — no Supabase Realtime, no WebSocket, no DB. Browser polls the GitHub commits API every N seconds; if HEAD changed, refetch tickets and re-render.

## Why
Supabase adds setup, an Edge Function, three migrations, a CLI install, and a project that pauses after a week of inactivity. The board's actual job is "show me the markdown files in this repo" — git is the source of truth. Polling GitHub directly removes the entire backend dependency and keeps the watching experience close-to-live (30-second freshness).

## Done when
- The `supabase-js` script tag and `client.channel(...)` subscription block are removed from the generated board HTML.
- The board polls `GET /repos/{owner}/{repo}/commits?sha=main&per_page=1` (or equivalent) every 30 seconds. If the HEAD commit SHA differs from the cached one, refetch tickets.
- Polling interval is configurable via `change-mate/config.json` field `poll_seconds` (default 30, minimum 10).
- Tickets are re-fetched by reading individual ticket files via the Contents API, OR by parsing a single tickets-index file the build step writes. Pick the simpler path (likely: re-rendering the embedded JSON the board already ships with, but refreshed via a server-rendered JSON endpoint baked into a small static file in the repo).
- Auth: anonymous polling for public repos. For private repos, browser sends the user's existing PAT (already in localStorage from Add Story setup).
- The "live" indicator in the header changes meaning: shows "polling" with a green dot when active, "offline" if rate-limited or auth fails.

## Desired output
A human opens the board on GitHub Pages. Within 30 seconds of any push to `main` (a ticket move, a new ticket, etc.), the board reflects the new state without a manual refresh. No backend was ever contacted.

## Success signals
- Adopters with public repos see the board work without setting up anything beyond the initial `setup.sh`.
- Adopters with private repos see it work with just their PAT.
- Push-to-board latency stays under ~35 seconds at default interval.
- GitHub API rate limit headroom — anonymous public-repo polling at 30s = 120 calls/hour, well under the 60/hour anonymous limit per IP for unauth'd... wait, that's tight. Authenticated polling at 30s = 120 calls/hour, well under the 5000/hour PAT limit.

## Failure signals
- Anonymous polling exhausts the 60/hour rate limit for adopters viewing public repos without a token.
- A burst of pushes triggers thrashing refetches.
- Browser tab left open overnight runs unbounded fetches; needs document.hidden check to pause polling when tab is backgrounded.
- Pages cache makes the polled response stale even when GitHub returns fresh data.

## Tests
- Unit (new): `tests/test_build.py` asserts the generated `board.html` no longer contains `supabase` / `supabase-js` / `realtime`.
- Unit (new): asserts the generated `board.html` contains the polling code (e.g. `setInterval` and `commits?sha=main`).
- Manual: open board, push a new ticket via git, wait ≤30s, confirm board updates without manual refresh.
- Manual: throttle network in devtools, confirm polling fails gracefully (header indicator shows "offline").
- Manual: switch tab away for 2 minutes, switch back — confirm polling resumes and refetches once.

## Notes
- **Alternative considered**: WebSocket via Cloudflare Workers/Durable Objects. Rejected — still requires deploying something. Polling at 30s is good enough for a kanban board people glance at.
- **Alternative considered**: GitHub webhooks → small Cloudflare Worker → SSE to browser. Cleaner UX, requires deploying a Worker. Out of scope for "kill all backends".
- **Alternative considered**: Drop polling entirely, rely on manual refresh. Rejected — too far from "live" feel.
- **Out of scope**: replacing cm-write (CM-058), removing the Supabase folder + docs (CM-059), live lock pulse (drop entirely as part of CM-059).
- **Risk**: anonymous public-repo polling rate limit. Mitigation: detect rate-limit response, back off + warn in the header indicator.
- **Confidence**: 80%. Main unknown is whether the board can re-render cleanly mid-session without a full page reload — the JSON is currently baked into the HTML at build time. May need a small static `tickets.json` that the build step writes separately so the polling code can refetch JSON only.
