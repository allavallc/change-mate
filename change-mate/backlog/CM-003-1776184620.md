# [CM-003] Ticket relationships (related / blocks / blocked-by)

- **Status**: open
- **Priority**: Medium
- **Effort**: M
- **Assigned to**: 
- **Started**: 
- **Completed**: 

## Goal
Let tickets reference other tickets so humans and agents can see dependencies and cross-ticket context at a glance.

## Why
Today "blocked" is just a folder — we lose *what* blocks *what*. Adding explicit relationships surfaces feature-set cohesion, reveals dependency chains, and makes the board useful beyond a flat list of cards.

## Done when
- Ticket format supports `Related`, `Blocks`, `Blocked by` fields (comma-separated CM-IDs)
- `CHANGEMATE.md` documents how agents set and read these fields
- Board renders relationships on the card (e.g. small "↔ CM-002" chips)
- Moving a ticket to `blocked/` requires a `Blocked by` value
- Unit tests in `build_lib.py` cover parsing: present, missing, malformed
- Round-trips cleanly through `build.sh` without data loss

## Notes
**Sequencing**: land CM-004 first so the new relationship fields slot into the updated ticket format in one pass.

**Desired output**: three new optional fields in every ticket, rendered as linked chips on the board.

**Success signals**:
- New tickets created after this ships include the new fields where applicable
- Clicking a relationship chip on the board scrolls/highlights the referenced card
- Legacy tickets without these fields still parse and render

**Failure signals / what to watch**:
- Orphaned references (CM-XXX listed but file not found) — parser should warn, not crash
- Cycles (A blocks B blocks A) — detect and flag on the board
- Board layout breaking when a ticket has many relationships

**Tests required**:
- Unit tests for each field: present, missing, malformed, multiple IDs
- Unit test for orphan detection
- Unit test for cycle detection
