# [CM-006] Supabase Edge Function `cm-write` (validated ticket writes via GitHub API)

## Goal
Build a Supabase Edge Function that validates a caller's write key, then creates/updates tickets in the GitHub repo on their behalf — keeping the GitHub PAT server-side only.

## Why
Today the "Add story" button on the board requires every user to paste their own GitHub PAT into localStorage. That is clunky, insecure, and blocks non-technical users. An Edge Function with the PAT stored as a Supabase secret is the right shape: one shared write path, per-person write keys, no tokens in the browser.

## Done when
- Edge Function `cm-write` deployed to Supabase
- Accepts: write key, ticket payload (title, goal, why, done-when, priority, effort, notes)
- Validates write key against `write_keys` table (hashed lookup)
- On success: creates the markdown file in `change-mate/backlog/` via GitHub API (authenticated with server-side PAT)
- On success: inserts a row into `ticket_events`
- On failure: returns structured error (`401` invalid key, `403` revoked, `422` bad payload, `500` GitHub API failure)
- Rate limit: per-key throttle (e.g. 10 writes/minute)

## Desired output
A callable HTTPS endpoint the board JS hits with a write key and a ticket payload. PAT never reaches the client.

## Success signals
- A valid write key + valid payload produces a new file in the repo and a `ticket_events` row
- Invalid/revoked keys are rejected with clear error codes
- GitHub PAT never appears in client bundle, browser network logs, or function output logs

## Failure signals
- PAT accidentally logged in function output
- Race condition on CM-ID assignment (two writes getting CM-009)
- Edge Function cold-start latency spiking add-story UX
- GitHub API rate limits under burst writes

## Tests
- Unit tests for key validation (valid / invalid / revoked / malformed)
- Unit tests for payload validation (missing fields, oversized notes)
- Integration test against a sandbox repo: full round-trip create
- Concurrency test: ten parallel writes get ten sequential CM-IDs, no collisions

## Notes
**Sequencing**: depends on CM-005 (needs `write_keys` + `ticket_events` tables). Blocks CM-007.

**Alternatives considered**:
- Frontend-direct GitHub OAuth flow → rejected; requires every user to register an OAuth app
- Server proxy outside Supabase (e.g. Cloudflare Worker) → rejected; introduces a second cloud dependency
