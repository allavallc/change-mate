# [CM-004] Reframe CHANGEMATE.md agent persona: senior technical PM, not senior dev

- **Status**: open
- **Priority**: Medium
- **Effort**: M
- **Assigned to**:

- **Completed**: 

## Goal
Rewrite the persona and working-style sections of `CHANGEMATE.md` so the agent operates as a senior technical product manager shaping features inside feature sets — not as a senior developer.

## Why
The current opening line ("You are a senior developer and project manager") pushes agents toward implementation voice. The real job at the ticket layer is PM work: crisp problem statements, desired outputs, success/failure criteria, and handoff notes that tell the developer what to watch. Developer tone belongs downstream, inside the build phase.

## Done when
- Opening persona line rewritten to "senior technical product manager creating features within feature sets for a project"
- Ticket intake questions expanded to require:
  - Desired output
  - Success signals
  - Failure signals / what to watch
  - Unit tests required (explicit prompt, not optional)
- Ticket file format gains new sections: `## Desired output`, `## Success signals`, `## Failure signals`, `## Tests`
- `build_lib.py` + `build.sh` parse and render the new sections
- Legacy tickets (without new sections) still parse and render correctly
- Unit tests cover both legacy and new ticket formats

## Notes
**Desired output**: a CHANGEMATE.md rewrite plus a parser update. Tickets created after this ships feel noticeably more PM-oriented and give developers clearer guardrails.

**Success signals**:
- Every new ticket created after this ships contains the four new sections
- Developers consuming these tickets do not need follow-up clarification on what "done" looks like
- Board displays new sections without layout regressions

**Failure signals / what to watch**:
- Parser crashes on tickets missing the new sections
- Board layout breaks on long success/failure bullet lists
- Agents drift back toward dev-voice because the rewrite is too subtle — language must be firm

**Tests required**:
- Unit tests for `parse_ticket` on legacy format (no new sections)
- Unit tests for `parse_ticket` on new format (all new sections present)
- Unit tests for partial new format (some new sections missing)
- Snapshot test of board HTML rendering for a new-format ticket

**Sequencing**: land this before CM-003 so relationship fields slot cleanly into the new format.
