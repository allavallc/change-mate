# [CM-012] Show active agent/assignee on board cards via live locks

- **Status**: open
- **Priority**: Medium
- **Effort**: S
- **Feature set**: feature-set-001-supabase-upgrade
- **Assigned to**: 
- **Started**: 
- **Completed**: 

## Goal
Board cards should display which agent or person is currently working on a ticket, using the `locks` table as the source of truth. When an agent claims a ticket (inserts a row into `locks`), the board should show their name on the card in real time.

## Why
Right now you can't tell who's working on what just by looking at the board. In a multi-agent setup where several Claude instances are working in parallel, this visibility is critical — it prevents duplicate work and lets the human see at a glance what's happening.

## Done when
- On board load, query the `locks` table and overlay the active agent indicator on any card that has a lock
- When a lock is acquired or released, the board updates in real time (via Supabase broadcast or polling)
- Active agent appears as a small crab icon (🦀) with a unique number or short name next to it — visually distinct from the static "Assigned to" field
- Each agent/person picks their own name when they first enter their write key (e.g. "🦀 crabFather", "🦀 Ada") — the name is stored in localStorage alongside the key
- Cards with active locks show a subtle visual differentiation (e.g. a colored left border or glow) so they stand out
- Works correctly when multiple agents are working simultaneously — multiple crabs on separate cards

## Desired output
A board where you can see a 🦀 with a name on a card and instantly know which agent is working on it, updated live as agents pick up and release tickets.

## Success signals
- Opening the board while an agent is mid-task shows their name on the correct card
- Lock release (agent finishes) removes the indicator within seconds
- Multiple simultaneous agents each show on their respective cards

## Failure signals
- Stale lock indicators that persist after an agent finishes
- Lock query failing silently and showing no indicators at all
- Performance degradation from polling the locks table too aggressively

## Tests
- Manual QA: start a task in one session, verify the board shows the lock
- Manual QA: complete the task, verify the indicator disappears
- Verify `locks` table RLS allows anon SELECT (currently it doesn't — may need a policy addition)

## Notes
**RLS gap**: the `locks` table currently has no SELECT policy for anon/authenticated — it's invisible to the board's anon key. This ticket needs to add a read-only policy: `CREATE POLICY locks_select_all ON public.locks FOR SELECT TO anon, authenticated USING (true);` plus `GRANT SELECT ON public.locks TO anon, authenticated;`. This is safe because lock data (ticket_id + agent name + timestamp) is not sensitive.

**Real-time options**: (a) Supabase Realtime `postgres_changes` on the `locks` table (requires Realtime enabled on the table), or (b) broadcast events from the Edge Function that acquires/releases locks (if one exists), or (c) poll on an interval. Option (a) is cleanest if Realtime is enabled.

**Crab identity**: the agent/person picks their own name when entering their write key (stored as `cm_agent_name` in localStorage). The crab + name badge ("🦀 crabFather") sits in the top-right of the card face, visible without expanding. The name is passed to the `locks` table `agent` field when a lock is claimed, so the board can display it on any browser.

**Color coding**: each crab gets a deterministic accent color derived from its name (pick from a small palette of 8–10 distinct colors). The card's left border uses the same color so you can scan the board and instantly group work by agent.
