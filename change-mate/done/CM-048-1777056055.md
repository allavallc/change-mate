## Goal
Rewrite the top of CHANGEMATE.md so any reader (human or LLM) finishes the first screen knowing change-mate is local dev tooling that must never enter a deploy artifact, code must not depend on change-mate paths, git is the intended sync layer, and change-mate files should be excluded from code-quality passes.

## Why
An adopter using change-mate inside hana-core (a production repo) reported that both humans and LLMs in the repo could not tell whether `change-mate-board.html` was a deployable asset or a dev artifact, so change-mate files drifted into WIP branches and caused deploy-pipeline worry. The feedback author's own TL;DR: "one sentence at the top of CHANGEMATE.md saying this is local dev tooling, not product, and must never deploy" is the single highest-leverage change available — everything else in this feature set flows from that.

## Done when
- CHANGEMATE.md has a new "Scope" section at the very top (before anything else) stating change-mate is dev-only tooling and must never appear in a deploy artifact.
- CHANGEMATE.md has a new "One-way dependency" rule stating change-mate knows about code but code must not reference change-mate paths or tickets.
- CHANGEMATE.md has a new "Sync modes" note stating git-sync is the default and documenting that adopters who want change-mate private per-machine can add `change-mate/` to `.gitignore` (with an explicit call-out that this disables collaboration).
- CHANGEMATE.md has a new "LLM guidance" block: do not lint/typecheck/format change-mate files as part of code-quality passes, exclude change-mate changes from repo-change summaries by default, assume `change-mate/` does not exist at deploy/build time.
- README.md "Files" table updated to tag each change-mate file as "dev-only tooling".
- No code change. Docs only.

## Desired output
A reader opening CHANGEMATE.md for the first time finishes the first screen knowing: what change-mate is, where it must not appear, how code and change-mate relate, whether they can run it privately, and how to treat change-mate files during code review. No inference required.

## Success signals
- Future adopter feedback does not repeat the "is this product code?" confusion.
- LLMs (Claude, Codex, other agents) reading the repo do not propose wiring code references to change-mate paths when drafting feature work.
- Solo-only users no longer ask whether they should commit `change-mate/` to a shared remote.

## Failure signals
- Scope statement drifts out of sync with actual behavior (e.g. a future ticket ships something that does deploy change-mate files — then the statement is a lie).
- "Do not lint/typecheck" guidance gets over-interpreted and adopters exclude change-mate from ALL tooling including basic repo hygiene (trailing whitespace, etc.).
- Local-only `.gitignore` guidance causes adopters to skip commits and then discover their tickets vanished on machine switch.

## Tests

- Manual: verify README.md files table labels change-mate files as dev-only.
- Existing pytest suite passes unchanged.

## Notes
- **Alternative considered**: split into four tiny tickets (one per feedback item). Rejected — all four changes touch the same CHANGEMATE.md preamble, splitting just creates edit churn and four board rebuilds.
- **Alternative considered (sync modes)**: build first-class support for local-only mode (no git sync, live-lock and Realtime still work per-user). Rejected as out of scope — the lock/Realtime design assumes shared ticket state. One-paragraph doc note is correct for v1; revisit if there's real demand.
- **Out of scope**: folder layout changes (CM-049), setup.sh deploy-ignore defaults (CM-050), workflow changes (CM-051).
- **Risk**: LLM guidance is advisory only — adopters' own linters will still run on `change-mate/` unless they exclude it themselves. CM-050 handles the mechanical exclusion.
- **Confidence**: 95%. Highest-ROI ticket in this feature set per the feedback author's own TL;DR.

## Resolution
Added three new top-of-file sections to `CHANGEMATE.md`: `## Scope` (scope statement as blockquote + one-way dependency rule), `## Sync modes` (git-sync default + local-only caveat), and `## For LLMs reading this file` (four bullet rules on lint/summary/build-time/ticket-ID treatment). Annotated the `README.md` Files table with `*(dev-only tooling)*` on each change-mate row. Full 77-test pytest suite passes unchanged. No code touched — docs only, as scoped.
