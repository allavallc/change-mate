## Goal
Move every change-mate artifact that currently sits at repo root (`CHANGEMATE.md`, `build.sh`, `build_lib.py`, `change-mate-board.html`, `change-mate-config.json`) into the `change-mate/` folder, so that the rule "anything under `change-mate/` is dev tooling" becomes mechanically true and `.dockerignore` / `.gcloudignore` exclusions collapse to one line.

## Why
The current scattered layout forces every adopter and every LLM to independently build a mental model of which files belong to change-mate and which are product. That mental model fails — see CM-048's underlying feedback. Consolidation makes the rule enforceable by tooling instead of tribal knowledge, and it's the enabler for CM-050's single-line deploy-ignore default.

## Done when
- `CHANGEMATE.md` lives at `change-mate/CHANGEMATE.md`.
- `build.sh` lives at `change-mate/build.sh`. It still runs from repo root (`bash change-mate/build.sh`).
- `build_lib.py` lives at `change-mate/build_lib.py`.
- Board is emitted at `change-mate/board.html`.
- `change-mate-config.json` lives at `change-mate/config.json`.
- `setup.sh` writes the CLAUDE.md import as `@change-mate/CHANGEMATE.md`, and installs the new layout for fresh installs.
- `setup.sh` detects the legacy layout (root-level `CHANGEMATE.md` + root-level `change-mate-board.html`) and offers an explicit migration step that moves the five files and rewrites the CLAUDE.md import. Migration is opt-in, not forced silently, and is idempotent.
- `.github/workflows/build-board.yml` updated: `run: bash change-mate/build.sh`, `git add change-mate/board.html`.
- `tests/` updated so every test references the new paths. `test_build.py` integration test runs the script from its new location.
- README.md, SETUP.md, docs/onboarding/README.md, CLAUDE.md references all updated to new paths. No stale `change-mate-board.html` or `change-mate-config.json` references remain in live code/docs.
- Generated board HTML has no hard-coded references to the old filenames.

## Desired output
After this lands, `ls` at the repo root of an adopter shows `change-mate/` and `setup.sh` — nothing else is change-mate's. Adding `change-mate/` to `.dockerignore` fully excludes change-mate from every deploy artifact. A new contributor opens the repo and instantly knows which folder is workflow tooling.

## Success signals
- `.dockerignore` exclusion for change-mate is one line (`change-mate/`), verified in a sample adopter repo.
- No adopter reports "change-mate files leaked into my deploy" post-ship.
- LLMs asked to summarize recent changes in an adopter repo cleanly bucket change-mate vs product commits by folder.

## Failure signals
- Existing adopters' boards break silently because their workflow still references `build.sh` at root. Mitigation: setup.sh migration path + release note.
- `bash change-mate/build.sh` breaks on Windows Git Bash paths. Mitigation: CI integration test.
- `build.sh` reads config by absolute path (e.g. `change-mate-config.json` as sibling) and the move breaks credential loading.
- Adopters with bookmarked `change-mate-board.html` URLs (GitHub Pages) hit a 404 until they update the link.

## Tests
- `tests/test_build.py`: run `bash change-mate/build.sh` from repo root, confirm `change-mate/board.html` is produced, confirm old `change-mate-board.html` is NOT produced.
- Unit: `parse_ticket` and `parse_feature_set` still work unchanged (pure functions, just moved).
- Manual: on a throwaway clone with the legacy layout, run setup.sh migration path — verify it moves all five files correctly and rewrites the CLAUDE.md import. Run migration again — verify no-op.
- Manual: `grep -r "change-mate-board.html" .` and `grep -r "change-mate-config.json" .` after the change — the only hits should be historical ticket bodies and changelog notes.
- Manual: open the generated `change-mate/board.html` in a browser and verify cards, Supabase live locks, and Add Story still function.

## Notes
- **Alternative considered**: keep the root layout but aggressively prefix files (`change-mate-build.sh` etc.). Rejected — doesn't solve the one-line deploy-exclude problem and leaves the mental model fractured.
- **Alternative considered**: keep `CHANGEMATE.md` at root, move the rest. Rejected — feedback explicitly calls out root-level CHANGEMATE.md as part of the "looks like product" problem. Keeping it out of the folder breaks the one-folder rule that makes everything else mechanical.
- **Alternative considered**: emit the board as `change-mate/index.html` so Pages serves it at `/change-mate/`. Attractive but out of scope for this ticket — stick with `board.html` and let a future ticket decide the public URL story.
- **Breaking change**: yes. Existing adopters must run migration. Ship with a visible changelog entry and tested setup.sh migration path.
- **Out of scope**: deploy-ignore defaults (CM-050), opt-in CI commits (CM-051), any change to ticket format.
- **Risk (highest)**: `setup.sh` runs remotely via `curl | bash`. Migration branch must be idempotent and must not destroy local uncommitted changes to any of the five files.
- **Confidence**: 75%. Mechanically obvious, but the breaking-change cost is real.
