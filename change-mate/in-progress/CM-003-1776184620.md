## Goal
Let tickets reference other tickets so humans and agents can see dependencies and cross-ticket context at a glance.

## Why
Today "blocked" is just a folder — we lose *what* blocks *what*. Adding explicit relationships surfaces feature-set cohesion, reveals dependency chains, and makes the board useful beyond a flat list of cards.

## Done when
- Ticket format supports `Related`, `Blocks`, `Blocked by` fields (comma-separated CM-IDs)
- `CHANGEMATE.md` documents how agents set and read these fields
- Board renders relationships on the card (small chips, cap + "+N more" overflow)
- Board auto-infers the inverse of `Blocks` → `Blocked by` at render time (not persisted in files)
- Moving a ticket to `blocked/` requires a `Blocked by` value
- Round-trips cleanly through `build.sh` without data loss

## Desired output
Three new optional fields in every ticket, rendered as chips on the card face, with the inverse edge inferred automatically. Build-time warnings for orphaned or cyclic references.

## Success signals
- New tickets created after this ships include the new fields where applicable
- Chips on CM-A say "Blocks: CM-B" and CM-B's card automatically shows "Blocked by: CM-A" without the user writing it twice
- Legacy tickets without these fields still parse and render
- Overflow displays cleanly on tickets with many relationships

## Failure signals
- Orphaned references (CM-XXX listed but file not found) — build logs a warning, chip is skipped
- Cycles (A blocks B blocks A) — build logs a warning, cards still render
- Board layout breaking when a ticket has many relationships — overflow cap prevents this
- Symmetric inference double-renders an edge (e.g. shows "Blocked by: CM-X" when CM-X file also explicitly lists "Blocked by" on the same target)

## Tests
- Parser unit tests: each field present / missing / single ID / multiple IDs / malformed
- Parser unit test: empty bullet value (just `**Related**:` with nothing after)
- Build-time test: orphan reference produces a warning in stderr but build exits 0
- Build-time test: cycle produces a warning but build exits 0
- Snapshot / manual: board card with all three relationships renders as expected chips

## Notes
**Design decisions agreed with user**:
- **Symmetry**: inferred at render time, not persisted. CM-005 writes `Blocks: CM-006`; CM-006's card auto-shows `Blocked by: CM-005` via the renderer walking the reverse map.
- **Placement**: all chips on card face, cap to 3 visible with "+N more" overflow beyond that.
- **Orphan / cycle handling**: warn in build log, skip the broken chip, keep building. Non-fatal.

**Plan**: see `tasks/todo.md` for the three-phase build.

**Alternatives considered**:
- Encode relationships only in feature-set files → rejected; loses ticket-level traceability and forces lookup detour
- Use a separate `relationships.json` index → rejected; duplicates source-of-truth from the markdown
- Require explicit two-sided edges (no inference) → rejected per user direction; doubles bookkeeping for no real gain
