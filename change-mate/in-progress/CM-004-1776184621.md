## Goal
Rewrite the persona and working-style sections of `CHANGEMATE.md` so the agent operates as a senior technical product manager shaping features inside feature sets — not as a senior developer. Add a reusable `product-manager` skill that owns ticket intake.

## Why
The original opening line ("You are a senior developer and project manager") pushed agents toward implementation voice. The real job at the ticket layer is PM work: crisp problem statements, desired outputs, success/failure criteria, and handoff notes that tell the developer what to watch. Developer tone belongs downstream, inside the build phase.

## Done when
- Opening persona rewritten to senior technical PM voice
- 6-question numbered intake retired; replaced with PM-skill-driven "draft first, ask second" flow
- Ticket format gains `## Desired output`, `## Success signals`, `## Failure signals`, `## Tests`
- Ticket format gains `**Feature set**` field
- New `skills/product-manager/SKILL.md` written from scratch (no external clone), wired into `setup.sh`
- `build_lib.py` parses new sections + feature_set; legacy tickets still parse
- `build.sh` renders feature set on card face, new sections on card open
- pytest harness + GitHub Actions CI runs on every PR
- CM-003 and CM-005–CM-009 backfilled into the new format
- Work split into 6 phases (≤5 files each), each with explicit user approval gate

## Desired output
The agent that intercepts "add a story about X" produces a fully-drafted, PM-grade ticket — including success/failure signals and test directives — instead of asking the user a numbered question list.

## Success signals
- New tickets created after this ships contain all four new sections without prompting
- Tickets carry a feature-set assignment at creation time
- Parser + renderer handle legacy and new formats with zero errors
- pytest runs green in CI

## Failure signals
- Parser crashes on malformed sections from older or hand-written tickets
- LLM drifts back into dev-voice on sub-tasks (skill prompt needs to be firm)
- Feature-set proposals feel random — skill needs clear heuristics for "is this a match"
- Board card face becomes visually crowded by the feature-set label
- Tests flake on CRLF/LF line ending differences (Windows ↔ CI) — addressed via `.gitattributes` in Phase 5
