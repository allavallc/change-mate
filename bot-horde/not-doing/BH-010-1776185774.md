# [BH-010] Second-LLM review pass on ticket drafts before user sees them

## Goal
Add an optional second-LLM review step that critiques a ticket draft (produced by the product-manager skill) before the draft is shown to the user — catching weak "done when" criteria, missing success/failure signals, or muddled goals.

## Why
The product-manager skill (BH-004) lets one LLM act as the PM and draft tickets autonomously. Drafts will occasionally be thin, vague, or self-consistent-but-wrong. A second LLM with a critic prompt — senior PM reviewer, or senior tech lead — can flag those issues before they reach the user, raising draft quality and cutting review rounds.

## Done when

### Config shape
- Config file migrates from `bot-horde-config.json` to `Horde of Bots-config.jsonc` (JSONC = JSON + comments). Existing JSON readers updated to strip comments before parsing. Works in any project, no new runtime dependencies.
- New block in the config:
  ```jsonc
  {
    // ...existing fields...
    "reviewer_llm_engaged": false,
    // Set to true to enable second-LLM review of ticket drafts.
    // When true, BOTH fields below must be filled in.
    "reviewer_llm": {
      "provider": "",   // e.g. "anthropic", "openai"
      "api_key": ""     // your provider key
    }
  }
  ```
- Off by default. Explicit opt-in. Setup wizard does NOT prompt for it — users flip the flag themselves.

### Behavior contract
- `reviewer_llm_engaged` absent or `false` → no review, no network call, no warning. Behavior identical to BH-004 baseline.
- `reviewer_llm_engaged` true AND both `provider` + `api_key` non-empty → review pass runs.
- `reviewer_llm_engaged` true AND `provider` or `api_key` missing/empty → **reject**: skip the review, print a one-line warning to stderr, append a one-line note to the draft the user sees ("⚠ reviewer config incomplete — skipping review"). Draft still proceeds normally.
- Any runtime error during review (network failure, invalid key, bad provider response, unknown provider name) → **log the error to stderr, include a one-line note in the draft, continue with the unreviewed draft**. Never crash the PM skill or the session.

### Review flow (when engaged and valid)
- Review output is either: "approved" (pass through) or "revise with: …" (draft rewritten exactly once — no loops).
- User sees the final (reviewed) draft. The review exchange is not shown unless they ask.

## Desired output
An optional critic pass that tightens drafts when turned on, and is a silent no-op when turned off. A misconfiguration is visible but non-fatal — it never blocks ticket creation.

## Success signals
- Engaged + valid: drafts arrive with tighter acceptance criteria and clearer risk callouts.
- Engaged + misconfigured: user sees "⚠ reviewer config incomplete" and the draft still lands. They fix the config at their leisure.
- Disengaged: behavior is identical to BH-004 baseline.
- Runtime error mid-review: user sees "⚠ reviewer errored — using unreviewed draft" and the draft still lands.

## Failure signals
- Latency spike on ticket creation when engaged (every draft now costs 2 LLM calls)
- Reviewer hallucinating requirements the PM never captured — draft drifts from user intent
- Infinite revise loops if not capped at one pass
- Any failure mode that crashes the PM skill instead of degrading gracefully

## Tests
- Unit: `reviewer_llm_engaged` absent → single draft pass-through unchanged, no warning surfaced.
- Unit: `reviewer_llm_engaged` false → identical to above.
- Unit: `reviewer_llm_engaged` true, `provider` empty → draft passes through with warning, no LLM call made.
- Unit: `reviewer_llm_engaged` true, `api_key` empty → same as above.
- Unit: engaged + valid + reviewer returns "approved" → draft unchanged.
- Unit: engaged + valid + reviewer returns "revise" → draft rewritten exactly once.
- Unit: engaged + valid + reviewer call throws → draft passes through with warning, no crash.
- Unit: JSONC parser handles: no comments, line `//` comments, block `/* */` comments, trailing commas disallowed (strict).

## Notes
**Sequencing**: depends on BH-004 (the PM skill must exist first). Low priority — ship only after the baseline PM skill has been in real use long enough to see where drafts actually get weak.

**Config format decision**: JSONC over plain JSON because the optional reviewer block benefits from inline instructions. JSONC over TypeScript because Horde of Bots must work in any project (Python, Go, static HTML, etc.) — forcing a TS runtime on non-TS users would break the "zero dependencies beyond git" promise.

**Error surface**: stderr + a one-line note on the draft the user sees. No separate log file. The user finds out about misconfiguration at the moment it matters — when they're about to approve a draft.

**Alternatives considered**:
- Always run reviewer (no toggle) → rejected; doubles cost and latency for every draft
- Self-review by the same LLM in a second prompt → rejected; same blind spots, less critical distance
- TypeScript config file → rejected; forces TS runtime on every consumer (bash, python, board HTML) and breaks the "any stack" promise
- Separate log file for reviewer errors → rejected; adds state the user has to know about; stderr + draft note reach the human when it matters

## Decision (2026-04-16) — parked as future idea
Parked before build. Scope ballooned during the pre-build interview:

- JSONC migration touches every config consumer (`build.sh`, `build_lib.py`, embedded board JS, `setup.sh`) — a structural change to the core of the project for an optional polish feature.
- The provider-agnostic reviewer requires an external Python script that calls arbitrary LLM APIs (since the PM skill lives inside Claude and can't call OpenAI/etc. directly). That's net-new tooling just for this flow.
- Together, Phase 1 (JSONC) + Phase 2 (reviewer script + skill wiring) + Phase 3 (docs) is a multi-day build for a feature that might turn out not to matter in practice.

**Preserved for future resurrection**: the design decisions above (JSONC over TS, opt-in config shape, graceful-degradation contract, external Python script for provider-agnostic review) all stand. When the baseline PM skill has been in real use long enough to show where drafts actually get weak, unpark this ticket and build against the existing spec.
