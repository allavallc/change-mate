## Goal
Extend the commit-message provenance convention in BOTHORDE.md with two new `Trigger:` actions: `accepted` and `rejected`. Same shape as the existing `claim`, `done`, `blocked`, etc. Lets `git log` reconstruct the full lifecycle of any user-facing ticket including its acceptance step.

## Why
fs-010 (BH-074) established `Model:` and `Trigger:` trailers as the audit trail. Without naming the new lifecycle actions, acceptance commits look like ordinary edits and the audit gap reopens. Adding two values preserves the contract.

## Done when
- BOTHORDE.md "Provenance trailers" section lists `accepted` and `rejected` alongside the existing actions
- Example commit messages added showing both shapes:
  ```
  BH-090: accepted

  Model: claude-opus-4-7
  Trigger: BH-090 accepted
  ```
  ```
  BH-090: rejected — failed step 3 (form did not validate)

  Model: claude-opus-4-7
  Trigger: BH-090 rejected
  ```
- BH-085 acceptance-tester skill produces these trailers by default
- Audit-query examples added: `git log --grep "Trigger: .* accepted"` and `Trigger: .* rejected`

## Desired output
A reader running `git log --grep "Trigger: BH-090"` sees the full lifecycle: `claim` → (work) → `in-review` → `accepted` (or `rejected` → `claim` → ... → `accepted`). One ticket, one chronological audit trail.

## Success signals
- Every accepted user-facing ticket has at least one matching `Trigger: BH-XXX accepted` commit.
- Rejection commits include a one-line rejection summary in the subject (not just the trailer) for git-log readability.

## Failure signals
- Trailer drift — bots use `Trigger: BH-090 reviewed` or `signed-off` instead. Mitigation: BH-085 hardcodes the action strings.
- Subject and trailer disagree (subject says accepted, trailer says rejected). Watch for this in code review.

## Tests
- Pure docs ticket. No code path; verification is a docs review.
- BH-085's tests indirectly cover trailer production (the skill writes them).
