## Goal
Capture the install-time questions a careful adopter asks (CLAUDE.md import semantics, public-repo exposure, Pages collisions, PAT security, polling rate limits, skill versioning, idempotency) into a single file that ships with the install. Both bots and humans read it on first contact, so the same questions don't have to be answered repeatedly.

## Why
A real adopter sent ~12 install-time questions before running setup.sh. Most have correct answers but those answers live nowhere in the repo today — they're in scattered tickets, conversations, and people's heads. Centralising them in `bot-horde/INSTALL-FAQ.md` (installed automatically by setup.sh per HB-061) makes the project self-documenting at the moment adopters most need it.

## Done when
- `bot-horde/INSTALL-FAQ.md` exists in the source repo.
- It covers, at minimum:
  - **CLAUDE.md import**: what it does, what autonomous actions the agent will take after import (commit/push ticket moves), how to scope or override.
  - **Public-repo exposure**: tickets are world-readable on public repos; workarounds = private repo, sanitize bodies, or use only for non-sensitive work.
  - **GitHub Pages**: the board is auto-served at `<site>/bot-horde/board.html` if Pages root is `main`; how to opt out via Pages source.
  - **PAT model**: where the PAT lives (browser localStorage), what scope it has (repo-wide Contents write — GitHub limit), security boundary (per viewer, per browser, never shared).
  - **auto_commit_board**: default `true`; what it controls (CI commits of `board.html`); override path.
  - **Polling rate limits**: 5000/hr per authenticated user, 60/hr unauth per IP; behavior under sustained failure (auto-disable after 5 misses).
  - **Skill upgrade**: where the global skill lives, how to upgrade, why it's global.
  - **Idempotency**: setup.sh is safe to re-run; what gets skipped vs touched.
  - **Non-interactive install**: env var or flag to skip prompts (after HB-062 lands).
  - **Python requirement**: build.sh needs Python 3.
- File is concise — one sentence per answer where possible, link to deeper docs for the rest.
- File is referenced from the README setup section ("After install, see `bot-horde/INSTALL-FAQ.md`").
- HB-061 wires setup.sh to download this file alongside BOTHORDE.md.

## Desired output
An adopter who clones Horde of Bots and runs setup.sh sees `bot-horde/INSTALL-FAQ.md` in their repo. They (or their agent) read it once and get the answer to every common install-time question without contacting the maintainer.

## Success signals
- "How does the CLAUDE.md import work?" / "Are tickets public?" / "Where does the PAT live?" stop showing up in adopter feedback.

## Failure signals
- The FAQ goes stale as the code evolves. Mitigation: tie updates to the tickets that change behavior (HB-061 / 062 / 063 / 064 / 065 each link to their FAQ section).

## Tests
- Manual: read the FAQ cold, confirm every question from the original feedback batch has a matching answer.
- Manual: cross-reference each answer against current code/docs; flag any that no longer match.

## Notes
- **Alternative considered**: append the FAQ to BOTHORDE.md. Rejected — BOTHORDE.md is loaded into every Claude Code session via @-import; growing it costs context budget. INSTALL-FAQ.md is opened explicitly by the agent or human when they need it.
- **Out of scope**: explaining the entire workflow (that's BOTHORDE.md's job). The FAQ is install-time only.
