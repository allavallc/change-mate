## Goal
Fix two pre-existing test-infrastructure bugs that survived the change-mate → bot-horde rename: the repo-root `conftest.py` adds the wrong path (so `py -m pytest <specific-test>` fails); and the regression test that walks every committed ticket globs `CM-*.md`, so post-rename it walks zero files.

## Why
Both bugs make the test setup pretend to work. Standalone path invocations fail with `ModuleNotFoundError: build_lib`, hiding why CI passes but local runs of a single file don't. The regression test claims to verify every committed ticket parses — but currently walks zero files, catching nothing. As fs-010 adds new schema fields (Verification, Failure-mode), that regression test is exactly what we need to catch breakage on real tickets.

## Done when
- `conftest.py` at repo root adds `bot-horde/` to `sys.path` (not just repo root) so `from build_lib import ...` resolves regardless of pytest invocation form
- `tests/test_parser.py::test_every_committed_ticket_parses_without_error` globs `BH-*.md` and asserts a non-zero ticket count walked (so an empty walk is caught)
- `py -m pytest tests/test_parser.py` passes standalone (currently fails)
- Full `py -m pytest` continues to pass

## Desired output
A contributor running pytest with any invocation form succeeds. The regression test actually verifies the committed ticket tree.

## Success signals
- `py -m pytest tests/test_parser.py` passes — local debugging works
- Schema breakage from fs-010 work (new Verification/Failure-mode field handling) gets caught against real tickets, not just synthetic fixtures

## Failure signals
- conftest path fix breaks unrelated Python module resolution — unlikely (no name collisions in `bot-horde/`)

## Tests
- `py -m pytest tests/test_parser.py` standalone → passes
- `py -m pytest` from repo root → all tests pass (count grows by 0 or 1)
- Verify empty-walk assertion fires: temporarily change glob back to `CM-*.md`; the count assertion should fail
