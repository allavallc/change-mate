## Resolution
Rewrote the browser-side `createStory()` to call GitHub's Contents API directly. Three new helpers: `nextTicketIdNumber()` scans `D.tickets` for the highest `CM-NNN` and returns max+1; `renderTicketMarkdown(id, payload)` mirrors the old `cm-write/ticket.ts` template (status open, priority/effort, optional feature set + assignee, Goal/Why/Done when/Notes sections); `utf8ToBase64(str)` does TextEncoder + btoa for Contents API content. Submit flow: PUT `/repos/{owner}/{repo}/contents/bot-horde/backlog/CM-NNN-<ts>.md` with the user's PAT, retry up to 5x on 422 (path conflict from race), surface 401/403 by clearing the PAT and reprompting setup, surface other errors via toast + console. Polling (BH-057) picks up the new file within 30s and reloads. cm-write Edge Function is no longer called by anyone.

## Goal
Replace the Supabase Edge Function `cm-write` with browser-direct calls to the GitHub Contents API. The Add Story button should write a markdown file to `bot-horde/backlog/` using the user's PAT — no server in between.

## Why
The Edge Function adds a deploy step, three Postgres migrations, a sequence + RPC, an SDK install, and a function to maintain. Its actual job is: validate a payload, claim an ID, render markdown, PUT to GitHub. Steps 1–3 happen fine in the browser. Step 4 happens fine in the browser. The middleman is unnecessary.

## Done when
- Add Story modal calls `GET /repos/{owner}/{repo}/contents/bot-horde/backlog` to list existing tickets. (Or queries the in-memory ticket list the board already has — preferred for simplicity.)
- Browser parses the highest existing CM-ID across backlog/in-progress/done/blocked/not-doing → next ID = max + 1.
- Browser renders ticket markdown locally using the same template cm-write uses today (port `ticket.ts` rendering logic to in-board JS).
- Browser PUTs to `https://api.github.com/repos/{owner}/{repo}/contents/bot-horde/backlog/CM-NNN-<timestamp>.md` with the user's PAT.
- On 422/409 (path conflict — race with another Add Story click), browser increments CM-ID and retries once.
- On 401/403, browser clears the cached PAT and prompts the setup modal.
- The `cm-write` URL and anon key placeholders in `build.sh` are removed.

## Desired output
A user opens the board, clicks Add Story, fills the form, clicks Create Story. Within ~2 seconds the file lands in the repo, CI rebuilds the board, polling (BH-057) detects the change and re-renders. No Supabase function was contacted.

## Success signals
- Add Story works for adopters who never deployed an Edge Function.
- Latency from click to "ticket appears in backlog" is comparable to current cm-write flow (PAT-direct GitHub call vs. cm-write → Postgres → GitHub call).
- No more 401/403/network errors stemming from a paused Supabase project.

## Failure signals
- Two simultaneous clicks race for the same CM-ID and one fails after retry. Mitigation: retry-on-conflict with a fresh ID scan.
- PAT scope insufficient — caller has only Read, can't PUT. UI must surface the GitHub error clearly.
- A push from another agent during the form-fill window means the ID we computed is now stale by the time we PUT. Same mitigation: retry once on 422.

## Notes
- **Alternative considered**: keep cm-write but rewrite it as a Cloudflare Worker. Rejected — still requires deploying something.
- **Alternative considered**: write the ticket via a GitHub Issue + Action that converts. Rejected — bigger architectural change, more moving parts than direct Contents API.
- **Out of scope**: removing the Supabase project / supabase folder (BH-059), polling code (BH-057).
- **Risk**: PAT exposure. The PAT lives in localStorage (already does today). Same security posture as current Add Story.
- **Risk**: ticket-ID race. The current cm-write uses a Postgres sequence for atomicity; browser-direct loses that. Mitigation is one retry on 422 — the CM-ID space is sparse (we're at BH-056, sequence supports billions), so a collision is rare and a single retry will succeed in practice.
- **Confidence**: 85%. The browser already has every input it needs; only thing being moved is the orchestration layer.
