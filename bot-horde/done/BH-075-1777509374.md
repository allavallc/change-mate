## Goal
Existing `Blocked-by:` field is already parsed and rendered. Add: (a) a "Ready only" filter toggle on the board that hides backlog/in-progress tickets whose `Blocked-by:` references a ticket not yet in `done/`; (b) validator rule (BH-076) refusing `done/` placement while `Blocked-by:` references unfinished tickets.

## Why
Today nothing prevents a bot from claiming a ticket whose dependencies aren't met. If BH-005 is `Blocked-by: BH-003` and BH-003 is in `backlog/`, a bot can grab BH-005 and burn cycles. The schema already encodes the dependency; we're not adding a feature, we're enforcing the field that already exists. The board filter helps humans + bots see only ready work; the validator catches premature done-transitions in CI.

## Done when
- Board filter bar gains a "Ready only" toggle (single checkbox, persists in `localStorage` under existing `hb_board_filters_v1` key)
- When toggle is on: tickets with at least one `Blocked-by:` reference that is **not** in `done/` are hidden from `backlog/` and `in-progress/` columns. `done/`, `blocked/`, and `not-doing/` columns are unaffected (they always show).
- Default toggle state: **off** — preserves current behavior for existing adopters; users opt in.
- Validator (BH-076) refuses `done/` placement if any `Blocked-by:` reference points to a ticket not in `done/`. Error message lists the unmet deps.
- BOTHORDE.md updated: agents check `Blocked-by:` before claiming; the validator will catch errors at build time
- `tests/test_build.py` covers: ready-only filter HTML output (toggle off → all visible; toggle on → blocked-by-unmet hidden)
- Validator test (in BH-076): done-transition with unmet deps fails

## Desired output
User toggles "Ready only" on the board. The backlog column shrinks to actionable work. CI catches accidental done-transitions of dependency-unmet tickets at PR time.

## Success signals
- "Ready only" becomes a default-on view for active contributors (their preference stored in localStorage).
- The validator rule catches at least one bad transition during the rollout period.

## Failure signals
- "Ready only" filter logic conflicts with the existing single-select Priority/Effort/Feature-set filters in the bar. Mitigation: the new toggle is a checkbox alongside the dropdowns, not in the dropdown — independent filter axis.
- Bots claim tickets via raw filesystem moves and bypass the filter view entirely. The filter is for humans; the validator is the enforcement.

## Tests
- `tests/test_build.py` — render board with mixed deps, assert that `Ready only` mode hides the right cards
- Validator test (in BH-076) — done ticket with unmet `Blocked-by:` fails with a clear error listing the unmet deps

## Notes
- **`Related:` is not a hard dependency** — the filter only acts on `Blocked-by:`. `Related:` continues to be informational.
- **Default off** is a deliberate gentle rollout; can be flipped later if adopters prefer ready-only as the default.
- The filter logic ('referenced ticket is not in `done/`') treats `not-doing/` as not-done. If a `Blocked-by:` reference is in `not-doing/`, the dependent ticket stays hidden when "Ready only" is on. This is correct: the dependency was abandoned, not satisfied.
