## Goal
Make setup.sh the place that prevents Horde of Bots from leaking into deploy artifacts by default — by writing (or appending to) the common deploy-exclusion files on install.

## Why
HB-048 tells readers that Horde of Bots must not enter deploys. HB-049 makes that rule mechanically enforceable with one folder. HB-050 closes the loop by actually applying the one-line exclusion in every adopter's repo at install time, so nobody has to remember. Adopters using Docker, Cloud Build, Fly.io, or Vercel would otherwise each have to read BOTHORDE.md, internalize the rule, and wire it themselves.

## Done when
- `setup.sh` detects the presence of `.dockerignore`, `.gcloudignore`, `.vercelignore` and for each that exists, appends `bot-horde/` as an exclusion if not already present.
- If an ignore file does not exist, setup.sh does NOT create it (don't surprise adopters with new tooling files; only modify what's already there).
- Appends are idempotent — re-running setup.sh does not duplicate entries.
- setup.sh prints a one-line confirmation per file touched (e.g. `✓ added bot-horde/ to .dockerignore`).
- `.gitignore` gets a *commented* marker line `# bot-horde/ is dev-only tooling; do not ignore unless using local-only mode (see bot-horde/BOTHORDE.md)`. This is documentation, not an exclusion.
- The existing `.gitignore` warning (fire if `Horde of Bots` is actually ignored) stays and is consistent with the new comment.

## Desired output
An adopter runs `setup.sh` in a repo that already has Docker. The next `docker build` excludes `bot-horde/` from the image. No configuration, no thought required. Verifiable with `docker run <image> ls /app/Horde of Bots` → not found.

## Success signals
- Adopters running Docker-based deploys see zero Horde of Bots files in their images post-ship.
- No issue reports of "Horde of Bots files ended up in my deploy".

## Failure signals
- setup.sh appends to an ignore file using a non-default format (e.g. `.gcloudignore` syntax differs subtly) and breaks the file.
- Append runs on re-install and duplicates entries (idempotency bug).
- Adopter using the local-only mode from HB-048 is confused when `.gitignore` comment contradicts their usage. Mitigation: comment references HB-048's doc section.
- Adopter has carefully tuned ignore files and resents the append. Mitigation: single clear line with a comment explaining why.

## Tests
- Integration (new): run setup.sh twice in a temp dir with a pre-existing `.dockerignore` — confirm `bot-horde/` is added once, not twice.
- Integration (new): run setup.sh in a temp dir with no ignore files — confirm nothing is created.
- Manual: verify `docker build` in a sample adopter repo excludes `bot-horde/` after install.
- Manual: verify `.gitignore` gets the commented explanation line, not an exclusion line.

## Notes
- **Alternative considered**: create missing ignore files (`.dockerignore` etc.) if they don't exist. Rejected — pushes adopters into tooling they may not use.
- **Alternative considered**: ship a single `Horde of Bots.ignore` file and have all tools read from it. Rejected — no tool supports that; would just end up syncing content.
- **Out of scope**: board / ticket content changes (HB-048/049).
- **Risk**: list of ignore files isn't exhaustive (Docker, GCP, Vercel covers most — misses Netlify, Fly.io source deploy, Cloud Run). Start with common ones, extend on demand.
- **Depends on HB-049**: if Horde of Bots files are still scattered at root, the one-line exclusion doesn't work. This ticket assumes the consolidated layout.
- **Confidence**: 90%.

## Resolution
Added a deploy-ignore block to `setup.sh` that runs after the CLAUDE.md import step. For each of `.dockerignore`, `.gcloudignore`, `.vercelignore` that already exists, appends `bot-horde/` as an exclusion (using `grep -qxF` for exact-line match to avoid duplicating on re-run, with newline-safety before append). Missing ignore files are not created. Separately, when `.gitignore` exists, a dev-only-tooling marker comment is added: `# bot-horde/ is dev-only tooling; do not ignore unless using local-only mode (see bot-horde/BOTHORDE.md)`. The existing `.gitignore` warning (fires when `Horde of Bots` is actively ignored) is preserved. Smoke-tested three paths: (1) existing `.dockerignore` gets a clean append and idempotent re-run skips; (2) temp dir with no ignore files leaves the dir untouched; (3) `.gitignore` marker comment lands correctly. All 77 tests still pass.
