## Goal
Add `pollSource` to `config.json` (`"github" | "none"`), surface a manual-refresh indicator when polling is off, and update README + BOTHORDE.md to stop implying Bot Horde works on any git host.

## Why
The board polls GitHub specifically. On `file://` it auto-disables silently. On GitLab/Gitea/self-hosted it just fails quietly. The README's "no vendor lock-in" framing is misleading. An explicit `"none"` mode + an honest README is cheaper than building real multi-backend polling and stops the false claim.

## Done when
- `config.json` schema gains `pollSource`, default `"github"`
- Allowed values: `"github"` | `"none"` (other backends explicitly out of scope — see Notes)
- `"none"`: polling JS is omitted; masthead shows `_(static — refresh manually)_`
- `"github"`: existing behavior unchanged; existing `file://` auto-disable still applies
- README — replace "no vendor lock-in" with "GitHub, with a clean off-switch off-platform"
- BOTHORDE.md sync mode docs reflect the same
- INSTALL-FAQ.md adds: "Q: does the board work on non-GitHub hosts? A: yes, but live updates are GitHub-only — set `pollSource: 'none'` and refresh manually."
- pytest: build with `"none"` → no polling JS, refresh indicator present; build with `"github"` (default) → existing behavior

## Desired output
A non-GitHub adopter reads the README, sees the GitHub-specificity up front, sets `pollSource: "none"`, and gets a working static board with no broken polling attempts in DevTools.

## Success signals
- Non-GitHub adopters complete setup without DevTools polling errors
- "Why doesn't live update work on GitLab?" becomes a non-question

## Failure signals
- Adopters read `"none"` as "no live updates ever, even on GitHub" — wording in config comment + docs should make the meaning clear

## Tests
- Build with `pollSource: "none"` → no `bh-poll` script in output, `_(static — refresh manually)_` indicator present
- Build with `pollSource: "github"` → existing polling unchanged
- Invalid value → build warns and falls back to `"github"`

## Notes
- **Out of scope**: GitLab/Gitea/Bitbucket polling backends. Both reads (commits API) and writes (Contents API for Add Story) are GitHub-shaped — multi-backend is a re-architecture, not a flag. Honest framing now; real backends only if a real adopter asks.
- **Why not auto-detect from remote URL?** Adopters may host the board separately from origin (GitHub Pages serving a GitLab repo). Explicit config is cleaner.
