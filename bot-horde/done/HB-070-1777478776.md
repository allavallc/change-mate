## Goal
Ship a permanent, view-only demo board at `demo/` so anyone landing on the repo can click in, look at a populated board, and play with the filter/sort UI without setting anything up.

## Why
The README + setup flow asks adopters to imagine what their board will look like once they have tickets. A live demo collapses that to one click. It also gives newcomers a feel for the brutalist styling, the walking robot, the per-bot crab colors, and the filter/sort UX without having to install anything.

## Done when
- `demo/data.json` contains the demo content as a single static fixture: 4 backlog tickets, 5 in-progress tickets (each with a different `assigned_to` so the per-bot crab + robot palette shows variety), 7 done tickets — all assigned across exactly 2 feature sets. **No parallel ticket tree under `demo/bot-horde/`** — the previous approach replicated the real repo's directory layout for fixture data, which the user (correctly) flagged as ridiculous.
- `bot-horde/build.sh` honors `HORDEOFBOTS_DEMO=1`. When set: skips the directory walk, loads tickets + feature_sets from `demo/data.json`, forces `demo_mode=true`, writes the rendered board to `demo/index.html`. Default behavior unchanged when the env var is absent.
- Demo mode emits a board with (a) empty `repo` config (disables polling — the demo never changes), (b) the "+ Add Story" button hidden so visitors can't be tricked into writing tickets to the real repo.
- `.github/workflows/Horde of Bots-rebuild-board.yml` builds the demo board alongside the real one on every push to main (gated on `demo/data.json` existing) and commits both via the existing `auto_commit_board` flow.
- README links to the live demo at `/demo/` (GitHub Pages serves `demo/index.html` as the index).
- Existing tests still pass; the demo build is verified locally.

## Desired output
A visitor opens the GitHub Pages URL for `/demo/`, sees a populated brutalist kanban with 4 columns of frozen tickets, two feature sets, walking robots on the in-progress cards, varied per-bot colors, and the working filter/sort bar. Clicking "+ Add Story" or trying to push state has no destructive effect (the button is hidden in demo mode; polling is off).

## Success signals
- README CTR to the demo URL is non-trivial (no formal metric — just a smell test that adopters click it).
- New issues / discussions reference the demo as the thing that sold them.

## Failure signals
- Demo content drifts from the production board's visual conventions (someone changes the real board's CSS but not the demo's, or vice versa) → the demo is misleading.
- A bug introduced into `build.sh` only manifests on the demo path (because demo_mode toggles output) and reaches main without anyone noticing → demo build needs at least one smoke check in CI or a test.
- A visitor finds the Add Story button still present in demo mode → write attempt fails or worse, succeeds against a public PAT and pollutes the repo.

## Tests
- Unit-ish: build.sh with `HORDEOFBOTS_ROOT=demo bash bot-horde/build.sh` produces `demo/bot-horde/board.html`. Verify by parse + grep.
- Build the demo locally; manually verify the 4/5/7 distribution renders, robots walk, filter/sort bar populates, both feature sets appear in the feature-set dropdown.
- Manually verify Add Story button is hidden when `demo_mode: true`.
- Real board build still works: `bash bot-horde/build.sh` (no env var) produces `bot-horde/board.html` unchanged.
- Existing pytest suite still passes — no test should reference the demo dir explicitly.

## Notes
- **Alternatives considered — separate "demo" branch**: rejected. A branch goes stale fast and Pages can only serve one branch by default. A subdirectory in main is simpler.
- **Alternatives considered — separate repo**: rejected. Cross-repo upkeep cost is high for a static showcase. Single repo, single main branch.
- **Alternatives considered — dynamic demo with live "play with it" mode**: rejected. Out of scope for v1. The demo is read-only; play = filter/sort interactions, not write.
- **Out of scope**: a "reset demo" mechanism. The demo content is frozen at commit time; if it gets edited, just revert.
- **Out of scope**: per-bot personality profiles or fake commit history. The bots are names + colors, nothing more.
- **Out of scope**: snapshotting the demo for the README screenshot. That's a separate ticket if pulled (the user wants to approve the demo first, then I capture).
- **Course-correction during build**: the first pass replicated the full ticket-tree layout under `demo/bot-horde/`, dumping 18 fixture markdown files into the repo. User pushed back — the demo content is fixture data, not a parallel project. Replaced with a single `demo/data.json` fixture file. Net repo cost dropped from 18 source files + a redirect to **2 files** (`demo/data.json` source, `demo/index.html` generated).
- **Confidence**: 90% (post-rework). Single static JSON + one env-var-gated branch in build.sh.
