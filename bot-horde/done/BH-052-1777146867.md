## Goal
Replace the existing border-glow pulse on in-progress cards (which is too subtle to read as "active") with a small 🤖 emoji that walks clockwise around the card perimeter. Reads instantly as "work is being done here" without being noisy.

## Why
The current `cm-active-pulse` (border color + box-shadow oscillation) is invisible to the user — they reported not seeing it. Humans need a clearer signal that a ticket is being worked on. A patrolling robot is universally legible as "active worker," uses zero dependencies (pure CSS + emoji), and only animates the four cards that actually need attention.

## Done when
- The `cm-active-pulse` keyframes and the `animation: cm-active-pulse ...` rule on `.card.status-inprogress` are removed.
- A new `.cm-robot` element is injected into in-progress cards via the cardHTML renderer.
- A new `cm-robot-walk` keyframe animation walks the robot clockwise around the card perimeter on a 12s loop, infinite, linear easing.
- `.card.status-inprogress` keeps a static green border so the status remains visually distinguishable from open/blocked cards.
- The robot has a soft green drop-shadow so it stays visible against both light and dark card backgrounds.
- pytest suite passes (existing 77 tests + a new assertion that `cm-robot` markup appears in HTML for in-progress tickets).

## Desired output
A human opens the board, sees one or more in-progress cards, and immediately notices small robot emoji walking around their borders. Multiple in-progress cards each have their own robot, each at different points in the loop (independent animations) — the board looks alive without being chaotic.

## Success signals
- User reports the animation is visible and reads as "in progress" without being told.
- No layout shift or scroll jank introduced.
- Animation continues smoothly when the board's Realtime updates fire.

## Failure signals
- Robot escapes the card boundary (positioning bug).
- Robot interferes with click targets on the card (catches pointer events it shouldn't).
- Animation stutters or pauses when other JS work happens.
- Emoji renders as a different glyph on some platform and breaks the metaphor.

## Tests
- Unit (existing): all 77 pytest tests still pass.
- Unit (new): `test_build.py` asserts the rendered HTML for an in-progress ticket contains `cm-robot`.
- Manual: open `bot-horde/board.html` in browser → confirm robot is visible, walks the four edges in order, completes loop in ~12s.
- Manual: dark mode toggle → robot still visible against dark card.

## Notes
- **Alternative considered**: typing-dots indicator. Rejected — less playful and doesn't tie to the existing visual vocabulary (🦀 crab badges already established a friendly-creature aesthetic).
- **Alternative considered**: shimmer sweep across card. Rejected — decorative, not semantic. Robot is clearer about "what is happening here."
- **Alternative considered**: rotate the robot to face the direction of travel. Skipped for v1 — keyframe complexity grows and the emoji's symmetric face makes the rotation read weirdly.
- **Out of scope**: animations on done/blocked/not-doing cards, sound, server-side push of "agent is typing" state.
- **Risk**: emoji glyph variation across OSes. 🤖 is widely supported (since Unicode 6.0); if it ever fails we fall back to a tiny inline SVG.
