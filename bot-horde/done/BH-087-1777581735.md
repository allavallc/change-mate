## Goal
Extend `bot-horde/validate.py` to enforce the new acceptance-loop schema and workflow rules. Without enforcement, `User-facing` rots, `## How to test` goes blank, and `Verification` gets self-set on user-facing work — the loop becomes decorative.

## Why
fs-010's lesson: schema fields without CI enforcement are noise. The acceptance loop adds three pieces of state (folder, field, section) and a workflow rule (Verification produced by tester, not dev). All four need machine-checkable rules or they will drift within weeks.

## Done when
- `validate.py` adds the following checks; collects all errors in one pass:
  - `User-facing` field present and one of `yes | no` on tickets created after the schema lands (legacy tickets without the field default to `no`, no error)
  - `## How to test` non-empty when `User-facing: yes` AND ticket is in `in-review/` or `done/`
  - Tickets in `in-review/` must have `Status: in-review`, `User-facing: yes`, and non-empty `## How to test`
  - Tickets in `done/` with `User-facing: yes` must have `Verification` set to `human-reviewed` or `bot-reviewed` (not `bot-claimed` or `tests-passed` — those are reserved for dev self-grading on non-user-facing work)
  - Tickets in `done/` with `User-facing: no` must have `Verification` set to `bot-claimed` or `tests-passed` (not the loop-only values)
- Errors print with the exact rule violated, file path, and corrective action
- `tests/test_validate.py` adds at least one fixture per new error class
- `tests/test_build.py` confirms `build.sh` exits non-zero when a fixture tree contains a malformed ticket against any new rule
- BOTHORDE.md "Validation" section gains a paragraph listing the new rules

## Desired output
A bot tries to push a `User-facing: yes` ticket to `done/` with `Verification: bot-claimed`. CI fails: `bot-horde/done/BH-NNN.md: User-facing: yes tickets cannot have Verification: bot-claimed — must come through in-review/`.

## Success signals
- At least one rule fires on a real PR during the rollout period (catching a drift attempt).
- `User-facing: yes` tickets consistently carry the right Verification by the time they hit `done/`.

## Failure signals
- Validator too strict — every legacy ticket without `User-facing` fails. Mitigation: legacy tickets default `User-facing: no` (don't error on absent field for tickets predating the schema; gate on file mtime or first-commit date if needed).
- Validator too lenient — schema drift goes uncaught. Confirm coverage by intentionally malforming a fixture ticket per rule and verifying the test catches it.

## Tests
- `tests/test_validate.py`:
  - `User-facing: maybe` → fails
  - `User-facing: yes` ticket in `in-review/` with empty `## How to test` → fails
  - `User-facing: yes` ticket in `done/` with `Verification: bot-claimed` → fails
  - `User-facing: no` ticket in `done/` with `Verification: human-reviewed` → fails
  - Legacy ticket missing `User-facing` field → passes (defaults to `no`)
- `tests/test_build.py` — build with a malformed fixture exits non-zero
