## Goal
Add a `**User-facing**: yes | no` field to the ticket schema. Default `no`. PM sets it at creation time on tickets that change something a user touches (UI, a flow, copy, behaviour visible to end users). The field is what triggers the acceptance loop — `yes` routes the ticket through `in-review/` before `done/`.

## Why
The acceptance loop has to know which tickets need a tester pass. Inferring from feature set is too coarse (a feature set bundles tickets of mixed user-facing-ness). A binary per-ticket flag is the simplest opt-in. Default `no` keeps internal/refactor work moving without friction.

## Done when
- BOTHORDE.md "Ticket file format" adds `**User-facing**: yes | no` between `Failure mode` and `Rejected by`
- `build_lib.parse_ticket` recognizes `**User-facing**:` and returns `user_facing` (string `"yes"` / `"no"`) on the ticket dict
- Default when field is missing or blank: treated as `"no"`
- Board renders a small `USER-FACING` badge on cards where `User-facing: yes` — distinct from priority/effort badges
- PM skill prompt updated: when drafting a ticket, ask "does this change something a user sees or interacts with?" → set `User-facing: yes` if so, else `no`
- BH-087 validator requires the field be one of `yes | no` if present

## Desired output
A ticket file containing `**User-facing**: yes` renders a visible badge on the board card. Bots reading the file get an unambiguous flag they can act on.

## Success signals
- New tickets land with the field set explicitly (yes or no), not blank.
- Most tickets are `no`; a meaningful subset is `yes`. If everything is `no`, the field is dead weight.

## Failure signals
- Bots default everything to `yes` to be safe → loop becomes mandatory and slow. Watch for >70% `yes` rate.
- Bots default everything to `no` to avoid the loop → field is decorative. Watch for <5% `yes` rate after the loop ships.

## Tests
- `tests/test_parser.py` — parse `User-facing: yes`, `User-facing: no`, missing field (defaults to `"no"`), invalid value (e.g. `"true"`) raises
- Validator test (in BH-087): ticket with `User-facing: maybe` fails

## Notes
- Field name is `User-facing`, not `Needs-acceptance` or `Reviewed`. Names what the work *is*, not what process it triggers.
- Out of scope: auto-detection of user-facing changes from the diff. Could come later as a v3.0 enhancement.
