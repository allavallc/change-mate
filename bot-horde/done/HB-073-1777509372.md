## Goal
Add a `**Failure-mode**:` field that is required when a ticket is in `blocked/` and absent/empty otherwise. Allowed values: `failed-tests | merge-conflict | context-exceeded | unmet-dep | needs-human`. Renderer shows the failure mode as a chip on blocked cards.

## Why
Today a ticket sitting in `blocked/` says nothing about *why* it's blocked. A reclaiming bot has to read Notes (which may not exist) to figure out if the blocker is "tests failing" (re-runnable), "context exceeded" (different bot can take it), or "needs a human decision" (action item for the user). Different blockers imply different next moves; making the mode explicit lets bots and humans triage at a glance.

## Done when
- Schema docs list the five allowed values with a one-line description of each
- `build_lib.parse_ticket` recognizes `**Failure-mode**:` and returns `failure_mode` on the ticket dict
- Board renders the failure mode as a chip on blocked cards — accent-bordered, mono uppercase
- BOTHORDE.md guidance: when a bot moves a ticket to `blocked/`, it must set `Failure-mode:`. Reclaiming bots read the field first to decide whether to take it.
- HB-076 validator requires `Failure-mode:` to be set when a ticket is in `blocked/`, and rejects invalid values
- `tests/test_parser.py` covers each value + a blocked ticket missing the field (validator test)

## Desired output
User glances at the blocked column. `needs-human` chips show what's waiting on them. `context-exceeded` and `failed-tests` chips show what another bot could pick up.

## Success signals
- Over time, the distribution of failure modes guides where to invest (lots of `context-exceeded` → smaller tickets; lots of `needs-human` → unclear specs).
- Reclaiming bots filter blocked tickets by failure mode.

## Failure signals
- Every blocked ticket gets `needs-human` because bots default to that. The field becomes useless.
- Mitigation: BOTHORDE.md documents specific guidance per mode so bots can pick correctly.

## Tests
- `tests/test_parser.py` — parse each of the five values
- Validator test (in HB-076): blocked ticket without `Failure-mode:` fails; blocked ticket with invalid value fails

## Notes
- **Out of scope**: introducing distinct folders per failure mode (e.g., `failed-tests/`, `merge-conflict/`). Field-on-card is the lightweight version; folders create mental + filesystem overhead and humans have to maintain them.
- Allowed values are deliberately small (5) and cover the actual failure shapes we've seen. Adding values later is cheap; removing them later is hard.
