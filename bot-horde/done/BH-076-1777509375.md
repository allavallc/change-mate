## Goal
Add `bot-horde/validate.py` that walks the ticket tree, parses every ticket, and asserts each one against the schema. Wire it into `build.sh` so CI fails on malformed tickets. This is the enforcement layer that makes the new fields from BH-071/072/073/075 trustworthy.

## Why
Today nothing checks that Status matches its folder, that dates parse, that referenced tickets exist, or that required fields are present. BH-060 sat in `backlog/` with `Status: in-progress` for two days because nothing caught it. With validation in CI, malformed tickets can't merge. As the schema gains fields (Verification, Failure-mode, Split-from), the only reasonable way to keep them honest is automated checking.

## Done when
- New file `bot-horde/validate.py` that:
  - Walks every ticket file in `backlog/`, `in-progress/`, `done/`, `blocked/`, `not-doing/`
  - Parses each with `build_lib.parse_ticket`
  - Asserts the following per ticket; collects all errors before exiting:
    - **Required fields present**: `Status`, `Priority`, `Effort`
    - **Status matches folder**: ticket in `in-progress/` has `Status: in-progress`, etc.
    - **Dates parseable**: `Started` and `Completed` (when present) are valid ISO dates or `YYYY-MM-DD HH:MM`
    - **`Assigned to` present** iff ticket is in `in-progress/` or `done/`
    - **`Completed` present** iff ticket is in `done/`
    - **`Verification:` present and valid** (value in `bot-claimed | tests-passed | bot-reviewed | human-reviewed`) iff ticket is in `done/`
    - **`Failure-mode:` present and valid** (value in `failed-tests | merge-conflict | context-exceeded | unmet-dep | needs-human`) iff ticket is in `blocked/`
    - **Reference resolution**: every ID in `Blocked-by:`, `Blocks:`, `Related:`, `Split-from:` resolves to a real ticket file
    - **Dependency satisfaction**: when ticket is in `done/`, every `Blocked-by:` reference must be in `done/` (BH-075 enforcement)
  - Prints all errors in a single pass with file path and line context
  - Exit non-zero if any errors found, zero if all pass
- `build.sh` runs `python3 bot-horde/validate.py` before generating HTML; build fails on validation error
- `.github/workflows/test.yml` already runs the build, so validation runs in CI by default
- New `tests/test_validate.py` with at least one fixture per error class
- BOTHORDE.md gains a short "Validation" section: tickets are validated in CI, here are the rules
- Backfill: any existing tickets that fail the new rules are fixed in the same PR (expected: very few — most current tickets predate the new fields, and the new fields are not retroactively required for legacy tickets)

## Desired output
A contributor opens a PR with a malformed ticket. CI fails with a precise message: `bot-horde/blocked/BH-099-1234.md: Failure-mode is required for blocked tickets but was not set`. They fix it; CI passes.

## Success signals
- At least one PR is caught by the validator during the rollout period.
- BH-060-style orphan-status tickets become impossible to land.

## Failure signals
- Validator too strict — legacy tickets fail and have to be backfilled across the board. Mitigation: legacy tickets without Verification/Failure-mode pass silently if the new field is absent on a ticket created before this feature set lands. Concretely: only validate `Verification:` on done tickets that were touched after the validator landed (use Started/Completed date as the gate, OR backfill all done tickets to `bot-claimed` in the same PR).
- Performance — walking ~30 tickets at build time is negligible (<100ms expected); flag if it becomes >500ms.

## Tests
- `tests/test_validate.py` covers each error class with a minimal fixture ticket:
  - Missing required field (Status, Priority, Effort)
  - Status/folder mismatch (Status: done in backlog/)
  - Unparseable date
  - Assigned-to missing on in-progress
  - Completed missing on done
  - Verification missing or invalid on done
  - Failure-mode missing or invalid on blocked
  - Unresolved reference (Blocked-by: BH-9999)
  - Done with unmet deps (Blocked-by: BH-003 where BH-003 is in backlog/)
- `tests/test_build.py` confirms `build.sh` exits non-zero when a tmp ticket tree contains a malformed ticket

## Notes
- **Build-sequence dependency**: this ticket is `Blocked-by: BH-071, BH-072, BH-073, BH-075` because it enforces fields/rules those tickets introduce. Build them first; this ticket lands last in the feature set.
- **No fix-it mode.** The validator only reports; it doesn't auto-fix. Auto-fix would mask schema drift instead of surfacing it.
- **Performance budget**: keep validation under 500ms on a tree of 100 tickets. Use simple file walks, no expensive parsing beyond what `build_lib.parse_ticket` already does.
