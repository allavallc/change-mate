## Goal
Introduce `bot-horde/in-review/` as a sixth ticket-state folder. Board renders it as a new column between `In Progress` and `Done` — same pattern as the other state folders today.

## Why
The acceptance loop (fs-013) needs a place where user-facing tickets pause after dev finishes and before they ship. Reusing existing folders (`blocked/`, `in-progress/`, `done/`) conflates states and breaks the columns-as-states model. A new folder is the smallest clean addition.

## Done when
- Empty directory `bot-horde/in-review/` exists with a `.gitkeep`
- `build.sh` walks `in-review/` alongside the other state folders
- `build_lib.parse_ticket` accepts `Status: in-review` as a valid value
- Board renders an `In Review` column between `In Progress` and `Done`
- BOTHORDE.md "Ticket file format" section adds `in-review` to the `Status` enum
- Validator (already covered by BH-087) accepts the new folder

## Desired output
A bot moves a ticket file into `in-review/` and on the next board build it appears in the new column. No other UI affordance changes.

## Success signals
- New column renders without breaking layout on narrow viewports.
- Filter dropdown selecting `In review` shows only tickets in that folder.

## Failure signals
- Six columns make the board feel cramped on smaller screens — flag if mobile/iPad views regress visibly.
- Existing tests that count five state folders break — update them in the same PR.

## Tests
- `tests/test_parser.py` — parse a ticket with `Status: in-review`
- `tests/test_build.py` — build runs against a fixture tree with an `in-review/` ticket; column appears in output HTML

## Notes
- Pure scaffolding ticket. No tester logic, no field changes — those live in BH-081/082/083.
- Out of scope: animating transitions between columns. Keep the visual style consistent with the existing brutalist column treatment.
