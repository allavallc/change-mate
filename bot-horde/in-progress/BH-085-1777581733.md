## Goal
Create a new Claude Code skill `acceptance-tester` (sibling to `product-manager`) that walks a bot through the tester role: pick a ticket from `in-review/`, read `## How to test`, execute the steps, decide approve or reject, and write the outcome back to the ticket. The skill enforces tester ≠ dev bot — it refuses to act on a ticket where the current bot identity matches the `Assigned to` of the dev work.

## Why
Without a skill, "bot plays tester" is a vague instruction. The skill turns it into a repeatable procedure with the same shape as `product-manager`: read context, execute, write artifact, commit with provenance. Self-approval prevention is the load-bearing part — without it, the loop is theatre.

## Done when
- New file `skills/acceptance-tester/SKILL.md` with `version: 1.0.0` in frontmatter (mirrors product-manager versioning)
- Skill contents:
  - When to invoke: bot picks up an `in-review/` ticket
  - Pre-check: read the most recent commit on the ticket — if its `Trigger:` line author matches the current bot's identity (or commit author email), refuse with a message and stop
  - Execute: read `## How to test`, run/observe the steps, decide
  - Approve path: move file to `done/`, set `Status: done`, set `Verification: bot-reviewed` (or `human-reviewed` if the user is driving), set `Completed`, commit with `Trigger: BH-XXX accepted`
  - Reject path: move file to `in-progress/`, set `Status: in-progress`, populate `Rejected by` / `Rejected` / `Rejection reason`, commit with `Trigger: BH-XXX rejected`
  - Reference the workflow in BOTHORDE.md "Acceptance loop" rather than re-stating it
- `setup.sh` installs `skills/acceptance-tester/` to `~/.claude/skills/acceptance-tester/` alongside `product-manager`
- `MANIFEST.json` includes the new skill file with version `1.0.0`
- BOTHORDE.md "Acceptance loop" section references the skill as the bot-tester entry point
- Tests: skill file parses (frontmatter valid, version present); setup.sh test confirms it installs

## Desired output
A bot says "test BH-090" and the skill takes over: identity check, read `## How to test`, execute, write outcome. Output is one commit with proper provenance trailers.

## Success signals
- Bot testers consistently use the skill rather than freelancing the workflow.
- No commit ever shows the same bot author on both the dev `Trigger: BH-XXX done` (or in-review handoff) and the `Trigger: BH-XXX accepted` line for the same ticket.

## Failure signals
- Identity check fails open — bot self-approves because the `Trigger:` author check is bypassable. Hardening: also check git config `user.email` against the dev commit's author email.
- Skill is bypassed entirely; bots write the loop transitions by hand. Detect via commits missing `Trigger: .* accepted|rejected` trailers in the in-review → done path.

## Tests
- `tests/test_setup.py` — skill installs to `~/.claude/skills/acceptance-tester/` when setup.sh runs
- Skill markdown parses without errors (frontmatter, version line)
- Manual: invoke the skill on a fixture ticket, confirm refusal when same-bot, confirm accept/reject paths produce the right file moves and commits

## Notes
- Keep the skill terse — match `product-manager`'s length and voice. Resist enumerating every edge case.
- Out of scope: a separate skill for human testers. The skill is for bots; humans following the same workflow read BOTHORDE.md directly.
- The "tester ≠ dev bot" check uses commit-author identity, which is project-configurable. Document the assumption in the skill.
