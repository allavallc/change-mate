## Goal
Document the three new workflow transitions in BOTHORDE.md: handoff (in-progress → in-review when `User-facing: yes`), acceptance (in-review → done), rejection (in-review → in-progress). Rejection reuses the existing `**Rejected by**` / `**Rejected**` / `**Rejection reason**` fields — no new schema.

## Why
The schema additions in BH-080/081/082 are inert without workflow guidance. Bots need to know *when* to move a ticket where, *who* writes what, and *how* approve/reject is recorded. Without the doc, the loop is theoretical.

## Done when
- BOTHORDE.md gains a new section "Acceptance loop" after "When work is complete"
- Section documents three transitions:
  - **Handoff (dev bot)**: when finishing a `User-facing: yes` ticket, instead of moving directly to `done/`, populate `## How to test`, move the file to `in-review/`, set `Status: in-review`, commit with `Trigger: BH-XXX in-review`
  - **Acceptance (tester, ≠ dev bot)**: read `## How to test`, execute, on pass move to `done/`, set `Status: done`, set `Verification: human-reviewed` (if human) or `bot-reviewed` (if bot), populate `Completed`, commit with `Trigger: BH-XXX accepted`
  - **Rejection (tester, ≠ dev bot)**: on fail move back to `in-progress/`, set `Status: in-progress`, populate `Rejected by` / `Rejected` / `Rejection reason`, commit with `Trigger: BH-XXX rejected`
- The non-user-facing path is unchanged: `User-facing: no` tickets go in-progress → done as today, dev bot self-sets `Verification: bot-claimed` or `tests-passed`
- "When work is complete" section gets a one-line note pointing to the new section for `User-facing: yes` tickets

## Desired output
A bot picking up an `in-review/` ticket finds clear instructions: how to identify it, what to read, what to write back, where to move the file. Same for a dev bot deciding whether to skip or enter the loop.

## Success signals
- New tickets follow the documented transitions without a session needing to ask "where does this go?"
- Rejection records carry enough information that the dev bot can fix without follow-up.

## Failure signals
- Dev bots skip `in-review/` and go straight to `done/` on user-facing tickets. Validator (BH-087) catches this.
- Tester bots approve their own work. BH-085 enforces the separation; this ticket only documents it.

## Tests
- No code in this ticket; it's pure docs. Verification is a docs review.
- BH-087 picks up the runtime enforcement of the rules described here.

## Notes
- The decision to reuse `Rejected by` / `Rejection reason` instead of inventing acceptance-specific fields keeps the schema flat. Rejection is rejection regardless of source.
- Out of scope: a `Reviewed by` field naming the tester on accepted tickets. Provenance trailers (BH-086) carry that information in the commit log; adding a ticket field would duplicate it.
